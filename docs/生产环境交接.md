## 6. 回滚规则

### 6.1 应用镜像回滚（仍使用新机）

- 新机保留了直接上一成功版本 `51c9fad` 的 API/Web 镜像。
- 仓库现有 `deploy/scripts/rollback.sh` 引用了已不存在的 Compose 服务 `web-static`，**禁止使用**。
- 需要回退镜像时，显式设置 `TAG=51c9fad` 执行 Compose，并在切换后核对 health、场景校验、
  容器重启数和公网链路。后续每次正常部署都会改变“上一版本”，操作前先检查本地镜像和 GHCR。

### 6.2 流量回滚到旧 VPS

旧机数据在切流时被冻结。**只要新机已产生任何用户写入，就绝不能直接启动旧 API 并切流，
否则会丢失或分叉新数据。** 回滚负责人必须：

1. 冻结新 API 写入并制作新机备份。
2. 将新机数据一致地回同步到旧机，比较文件数、字节数、内容和权限清单。
3. 启动旧 API并验证健康。
4. 在 `dmitsvr` 原子安装 `edge-old-origin.conf`，先 `nginx -t` 再 reload。
5. 将 GitHub `VPS_HOST` 同步恢复为旧 VPS，避免后续 CI 继续部署到错误主机。
6. 验证公网、数据和 CI 后再解除冻结。

一次性迁移 SSH key 已从两端删除。需要回同步时应重新建立来源受限的临时凭据，完成后立即移除。

## 8. 恢复策略决定

用户已于 2026-07-11 明确决定：Grandtianfu 不需要持续异地备份。当前策略为
`migration-snapshot-only`，异地备份周期和隔离恢复演练不再是观察期或旧机退役门禁。
第 5 节记录的两份冻结归档只作为本次迁移与回滚证据，不构成持续恢复服务。

不要 enable 仓库的 `grandtianfu-backup.timer`。除了策略不要求外，该实现本身也不能直接用于生产：

- unit 使用不存在的 `deploy` 用户和 `/srv/grandtianfu/deploy` 路径，实际路径是 `/opt/grandtianfu`；
- 目标机没有 `rclone`、remote 或 `BACKUP_REMOTE`；
- 宿主机旧脚本会漏掉 `artifacts/uploads`；
- 当前 restore test 即使内容 diff 失败也会返回成功，且不验证权限、ACL/xattr；
- 备份脚本直接 tar 活目录，没有一致性快照策略，也没有可靠失败告警和保留策略。

如果未来恢复策略发生变化，应把它作为新的运维变更重新设计和验收，不能把当前模板直接启用。
