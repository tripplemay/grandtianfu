# 项目基线与多方案工作流 — 产品交互规格

> 状态：产品决策已确认，可进入技术设计与实施  
> 版本：v1.0  
> 日期：2026-06-30

## 0. 已确认决策

以下决策为后续实现基线，不再作为开放问题：

1. 已确认户型不可原地修改；调整户型必须创建新版本。
2. 底层 `default` 在界面显示为“初始方案”，不具有特殊业务地位。
3. 当前户型版本默认只展示所属方案；旧版本方案进入“历史版本”。
4. 已确认方案不可原地解锁；“继续调整”必须自动复制为一套新草稿方案。
5. 户型版本被新版本替代后，禁止继续创建方案或生成新成果，只允许查看和导出。
6. 空房照片版本化和实拍效果图链路不在本批开发中，作为下一批独立实施。

## 1. 目标

将当前混合在一起的“户型编辑、家具方案、效果图生成”拆成两个明确作用域：

1. **项目基线**：先确定户型和空房资料，作为长期稳定的共同基础。
2. **软装方案**：在同一户型版本下创建、编辑、比较多套独立方案。

用户的主要工作方式为：

> 先确认户型，再长期创建、迭代和比较多套软装方案。

## 2. 核心原则

1. 一个项目可以有多个户型版本，但同一时间只有一个“当前户型版本”。
2. 已确认的户型版本只读，不能直接覆盖修改。
3. 修改已确认户型必须从当前版本创建一个新草稿版本。
4. 每套软装方案必须绑定一个明确的户型版本。
5. 家具、预览图、AI 效果图等方案成果不得跨方案混用。
6. 默认只展示当前户型版本下的方案，旧版本方案进入历史入口。
7. `default` 仅作为底层兼容 ID，界面统一显示为“初始方案”。
8. 页面和导航必须持续携带当前户型版本与方案上下文，不能静默回退。

## 3. 数据作用域

| 层级 | 数据 | 说明 |
| --- | --- | --- |
| Project | 项目名称、地址、创建时间、当前户型版本 | 项目级信息 |
| BaselineVersion | 房间、墙体、门窗、空间尺寸、空房照片、校验结果 | 所有绑定方案共享 |
| FurnitureScheme | 名称、风格意向、家具清单与位置、状态、首选标记 | 每套方案独立 |
| SchemeArtifact | 平面图、轴测图、AI 效果图、实拍效果图 | 必须绑定方案 |
| GenerationJob | 模型、输入、状态、警告、费用、错误 | 一次生成任务记录 |

“项目基线不变”表示它是方案共享的唯一来源，不表示永远不可演进；演进通过新版本完成。

## 4. 领域结构

```text
Project D
├── Baseline V1（superseded，只读）
│   ├── 初始方案
│   └── 现代简约方案
└── Baseline V2（current，已确认）
    ├── 初始方案
    ├── 现代轻奢方案
    └── 原木方案
        ├── furniture
        ├── plan / axon previews
        ├── AI renders
        └── real-room renders
```

## 5. 状态模型

### 5.1 户型版本状态

```text
draft ──确认──> confirmed ──新版本确认──> superseded
```

| 状态 | 是否可编辑 | 是否可创建方案 | 说明 |
| --- | --- | --- | --- |
| draft | 是 | 否 | 正在编辑和校验 |
| confirmed | 否 | 是 | 当前有效户型版本 |
| superseded | 否 | 否 | 历史版本，仅供查看和追溯 |

约束：

- 一个项目最多有一个 `confirmed` 当前版本。
- 新版本草稿创建期间，原确认版本仍保持当前状态并继续可用。
- 新版本确认后，原确认版本转为 `superseded`。
- 不允许删除已经产生方案或成果的户型版本。

### 5.2 软装方案状态

```text
draft ──确认──> confirmed ──归档──> archived
   └────────────────归档──────────> archived
```

| 状态 | 是否可编辑 | 是否参与默认比较 | 说明 |
| --- | --- | --- | --- |
| draft | 是 | 是 | 正在调整 |
| confirmed | 否 | 是 | 已定稿方案；继续调整时复制为新草稿 |
| archived | 否 | 否 | 历史保留 |

“首选”使用独立布尔标记，不作为生命周期状态：

- 同一户型版本最多一套首选方案。
- 设置新首选时，原首选自动取消。
- 首选方案仍可以是 `draft` 或 `confirmed`，但界面提示用户优先确认。
- 已确认方案不允许转回 `draft`。
- 从已确认方案继续调整时，创建一套 `source=duplicate` 的新草稿方案；原方案及成果保持不变。

### 5.3 AI 任务状态

```text
queued → running → done
                 ↘ error
```

任务完成后才写入方案成果历史。任务切换页面后继续执行，但结果必须归档到提交时绑定的方案，不能写入用户后来切换到的方案。

- 家具布置

### 6.3 持续上下文

进入项目后，顶部固定显示：

```text
D 户型 / 户型 V2 / 现代轻奢软装方案
```

其中：

- 项目名称可返回项目概览。
- 户型版本可打开版本信息，但不能在方案页面直接切换到不兼容版本。

没有进入具体方案时显示：

```text
D 户型 / 户型 V2 / 方案中心
```

## 7. 主工作流

### 7.1 新项目

1. 创建项目。
2. 自动创建户型版本 `V1 draft`。
3. 导入或编辑户型。
4. 执行空间、门窗、重叠和数据完整性校验。
5. 下一批可上传空房照片并关联房间；本批不提供该业务操作。
6. 点击“确认户型”。
7. 展示确认影响说明。
8. 用户二次确认后，将 V1 锁定为当前版本。
9. 进入方案中心。

### 7.2 创建方案

方案中心提供：

- AI 生成候选方案
- 创建空白方案
- 复制已有方案
- 从历史版本迁移方案

AI 生成始终创建新方案，不覆盖任何现有方案。

如果项目在引入新模型前已有根级家具数据，则首次进入时将其显示为“初始方案”。

### 7.3 编辑和生成成果

1. 从方案中心进入一套方案。
2. 调整家具布置。
3. 保存方案。
4. 查看平面图和轴测图。
5. 生成 AI 效果图。
6. 下一批可上传或选择空房实拍，生成实拍效果图；本批流程到 AI 轴测效果图结束。
7. 返回方案中心或进入方案对比。

所有成果历史均显示当前方案名称和户型版本。

### 7.4 方案对比

1. 在方案中心勾选 2–3 套同户型版本方案。
2. 点击“对比方案”。
3. 对比页采用统一视图模式并排展示。
4. 用户可切换：
   - 家具平面图
   - 轴测方案图
   - AI 效果图
   - 实拍效果图
5. 某方案缺少对应成果时显示明确空状态和“去生成”入口。
6. 用户可以设置首选、打开方案或退出对比。

禁止比较不同户型版本的方案，避免空间基础不一致导致误判。

### 7.5 修改户型

1. 在当前确认版本上点击“创建新版本”。
2. 复制当前户型形成 `V2 draft`。
3. V1 继续作为当前版本，已有方案和成果不受影响。
4. 用户编辑和校验 V2。
5. 点击“确认并启用 V2”。
6. 系统提示：
   - V2 将成为当前户型。
   - V1 及其方案转入历史入口。
   - 旧方案不会自动迁移。
7. 用户确认后切换当前版本。
8. 用户可从 V1 选择方案复制到 V2。
9. 复制后的方案状态为 `draft`，执行家具房间映射和越界校验。

## 8. 页面规格

### 8.1 项目概览

展示：

- 当前户型版本及确认状态
- 当前版本方案数量
- 首选方案
- 最近生成成果
- 待处理警告

主要操作：

- 户型未确认：“继续完善户型”
- 户型已确认：“进入方案中心”
- 有首选方案：“查看首选方案”

### 8.2 户型基线

草稿状态：

- 显示完整编辑工具。
- 显示校验结果和问题定位。
- 主按钮为“确认户型”。

已确认状态：

- 默认只读展示。
- 明确显示“已锁定，所有当前方案基于此版本”。
- 编辑入口替换为“创建新版本”。
- 禁止使用普通保存覆盖原版本。

### 8.3 空房照片

本节为下一批实施范围，本批仅保留导航与数据模型扩展位：

- 照片绑定户型版本，不绑定软装方案。
- 每张照片可以标注房间、拍摄方向和备注。
- 已被 AI 任务使用的照片允许删除，但必须二次确认并说明历史成果不会删除。
- 新户型版本默认复制照片引用，用户可重新标注。

### 8.4 方案中心

默认只展示当前户型版本的未归档方案。

每张方案卡展示：

- 方案名称
- 户型版本
- 状态与首选标记
- 风格意向
- 家具数量
- 效果图数量
- 最近更新时间
- 最新成果缩略图

操作：

- 打开方案
- 对比勾选
- 复制
- 重命名
- 确认
- 从已确认方案创建调整副本
- 设置首选
- 归档

页面提供“查看历史版本方案”入口，历史方案不混入当前列表。

### 8.5 方案工作区

顶部始终显示当前方案选择器。

规则：

- 页面切换必须保留方案上下文。
- 切换方案时清空旧方案瞬时状态并重新加载。
- 过期请求不得覆盖当前方案。
- 保存和生成操作必须在按钮附近显示目标方案名称。
- 方案已确认时只读。
- 点击“继续调整”后自动复制为新草稿，并跳转到新方案；不得修改原方案状态或内容。
- 历史户型版本下的方案只允许查看和导出，不允许复制以外的写操作；迁移到当前版本走专用流程。

### 8.6 方案对比

- 默认最多三列。
- 所有列使用相同缩放和视图模式。
- 每列固定展示名称、状态、首选标记和主要操作。
- 支持将任一方案设为首选。
- 不提供“应用方案”操作，避免产生覆盖其他方案的歧义。

## 9. 关键交互文案

### 9.1 确认户型

标题：`确认并锁定户型 V1？`

说明：

> 确认后，本版本将作为软装方案的共同空间基础，不能直接覆盖修改。后续调整需要创建新的户型版本。

### 9.2 创建新版本

标题：`从户型 V1 创建新版本`

说明：

> 系统将复制当前户型形成 V2 草稿。V1 及其所有方案和效果图保持不变。

### 9.3 启用新版本

标题：`确认并启用户型 V2？`

说明：

> V2 将成为当前户型，V1 及其方案进入历史版本。旧方案不会自动迁移。

### 9.4 确认方案

标题：`确认“现代轻奢软装方案”？`

说明：

> 确认后方案将锁定为只读，已有成果继续保留。如需调整，系统会复制一套新的草稿方案，原方案不会改变。

### 9.5 继续调整已确认方案

标题：`基于“现代轻奢软装方案”创建调整副本？`

说明：

> 系统将复制当前家具布置和风格设置，创建一套新的草稿方案。原定稿方案及其效果图保持不变。

默认新名称：

```text
现代轻奢软装方案 - 调整版
```

## 10. URL 与上下文规则

概念路由：

```text
/studio/projects/D/overview
/studio/projects/D/baseline
/studio/projects/D/baseline/photos
/studio/projects/D/baseline/versions
/studio/projects/D/schemes
/studio/projects/D/compare
/studio/projects/D/editor?scheme=<scheme_id>
/studio/projects/D/gallery?scheme=<scheme_id>
/studio/projects/D/render?scheme=<scheme_id>
```

MVP 可继续通过 query 传递 `scheme_id`，但必须满足：

- 所有方案内导航保留 `scheme`。
- 不允许通过隐式服务端“当前方案”决定操作目标。
- URL 没有 `scheme` 时，只有方案中心可以正常进入；方案工作页必须引导用户选择方案，不能静默使用 `default`。
- `baseline_version` 由方案绑定关系确定，不由前端自由拼接。

## 11. 历史与兼容

### 11.1 `default` 兼容

- 底层保留 `default` ID，避免破坏已有数据和接口。
- 界面统一显示“初始方案”。
- 初始方案与其他方案采用相同卡片、状态和操作。
- 不再通过缺失 `scheme_id` 隐式表示初始方案。

### 11.2 旧效果图

- 多方案上线前产生的项目级效果图归入“初始方案”。
- 旧文件路径保持可访问。
- 历史列表合并时按记录 ID 或 URL 去重。

### 11.3 旧户型数据

- 现有单一 `geometry.json` 迁移为 `Baseline V1 confirmed`。
- 已有方案绑定到 V1。
- 迁移过程不得改写几何内容或家具内容。

## 12. 异常与边界

- 当前没有已确认户型：禁止创建方案。
- 草稿户型校验失败：禁止确认，并提供问题定位。
- 方案对应户型已进入历史：允许查看，不允许继续生成新成果；需要先迁移到当前版本。
- 历史户型版本下禁止创建空白方案、AI 生成候选、复制为同版本方案或生成新成果。
- 已确认方案禁止保存家具、重命名关键设计内容或直接转回草稿。
- 已确认方案需要调整时，只能复制为当前户型版本下的新草稿方案。
- 方案切换期间：显示加载状态，不保留上一方案图片冒充当前结果。
- AI 任务执行中切换方案：任务继续，结果写回原方案。
- 删除或归档方案：不得删除已有成果文件；后续由独立清理策略处理。
- 户型版本存在方案或成果时：禁止物理删除。

## 13. 实施边界

### 13.1 本批开发

- 建立户型版本模型。
- 将现有户型迁移为 V1。
- 实现确认锁定与创建新版本。
- 调整导航和持续上下文。
- 将 `default` 显示为“初始方案”。
- 当前版本与历史版本方案分组。

第二阶段：

- 方案状态、首选和归档。
- 2–3 套方案并排比较。
- 旧方案迁移到新户型版本及重新校验。

### 13.2 下一批独立开发

- 空房照片版本化。
- 实拍效果图链路。
- 更完整的成果交付和清理策略。

## 14. 验收标准

1. 用户无法直接编辑已确认户型。
2. 用户可以从已确认户型创建新草稿版本。
3. 新版本确认前，当前版本和现有方案持续可用。
4. 每套方案都能追溯到明确的户型版本。
5. 当前方案名称在编辑、预览和效果图页面持续可见。
6. 页面切换后不会丢失方案上下文。
7. 切换方案不会短暂展示上一方案的家具或效果图。
8. 默认列表不混入历史户型版本的方案。
9. 用户可以进入历史入口查看旧版本、旧方案和旧成果。
10. 界面不再将 `default` 暴露为特殊业务概念。
11. AI 生成只创建新方案或新成果，不覆盖其他方案。
12. 默认方案的旧效果图迁移后仍可查看。
13. 不同户型版本的方案不能进入同一对比。
14. 当前版本下可以选择最多三套方案进行统一视图比较。
15. 已确认方案不能原地转回草稿。
16. “继续调整”会创建新草稿方案，并保留原方案及成果。
17. 历史户型版本禁止新增方案和生成成果。
18. 空房照片与实拍效果图入口在本批可以显示“下一阶段”，但不得提供未完成的可点击业务操作。

## 15. 建议存储结构

当前项目采用文件存储，本批继续沿用，不引入数据库。

```text
data/projects/D/
  project.json
  geometry.json                         # 旧接口兼容镜像，迁移期保留
  furniture.json                        # 旧 default 兼容镜像，迁移期保留
  baselines/
    v1/
      meta.json
      geometry.json
      validation.json
    v2/
      meta.json
      geometry.json
      validation.json
  schemes/
    default/
      meta.json                         # baseline_version_id=v1
      furniture.json
      renders.json
    scheme_ai_xxx/
      meta.json                         # baseline_version_id=v1
      furniture.json
      renders.json
```

`project.json` 建议结构：

```json
{
  "id": "D",
  "name": "阅天府 D 户型",
  "current_baseline_version_id": "v1",
  "next_baseline_version": 2,
  "created_at": "2026-06-30T00:00:00Z",
  "updated_at": "2026-06-30T00:00:00Z"
}
```

`baselines/v1/meta.json`：

```json
{
  "id": "v1",
  "status": "confirmed",
  "source_version_id": null,
  "created_at": "2026-06-30T00:00:00Z",
  "confirmed_at": "2026-06-30T00:00:00Z",
  "superseded_at": null
}
```

方案 `meta.json` 新增：

```json
{
  "baseline_version_id": "v1",
  "status": "draft",
  "preferred": false,
  "archived_at": null
}
```

约束：

- 版本 ID 使用项目内单调递增的 `v1`、`v2`，便于界面理解。
- 文件写入继续使用同目录临时文件加 `os.replace`，保证原子性。
- 确认新版本涉及多个文件时必须加项目级锁，并先写新状态，最后切换 `project.json` 当前指针。
- 旧根级 `geometry.json` 始终镜像当前确认版本；草稿编辑不能提前覆盖它。
- 旧根级 `furniture.json` 继续镜像“初始方案”，仅用于兼容旧接口。

## 16. 后端 API 草案

### 16.1 项目与户型版本

```text
GET  /api/projects/{project}
GET  /api/projects/{project}/baselines
GET  /api/projects/{project}/baselines/{version}
GET  /api/projects/{project}/baselines/{version}/geometry
POST /api/projects/{project}/baselines
POST /api/projects/{project}/baselines/{version}/save-geometry
POST /api/projects/{project}/baselines/{version}/validate
POST /api/projects/{project}/baselines/{version}/confirm
```

创建新版本：

```json
{
  "source_version_id": "v1"
}
```

确认版本必须在服务端同一临界区完成：

1. 确认目标版本仍为 `draft`。
2. 重新执行几何校验。
3. 将原当前版本改为 `superseded`。
4. 将目标版本改为 `confirmed`。
5. 更新 `project.json.current_baseline_version_id`。
6. 更新根级 `geometry.json` 兼容镜像。

### 16.2 方案

```text
GET  /api/projects/{project}/schemes?baseline_version_id=v1&include_archived=false
POST /api/projects/{project}/schemes
GET  /api/projects/{project}/schemes/{scheme}
PATCH /api/projects/{project}/schemes/{scheme}
POST /api/projects/{project}/schemes/{scheme}/confirm
POST /api/projects/{project}/schemes/{scheme}/adjust
POST /api/projects/{project}/schemes/{scheme}/archive
POST /api/projects/{project}/schemes/{scheme}/set-preferred
POST /api/projects/{project}/schemes/{scheme}/migrate
```

`adjust` 请求：

```json
{
  "id": "scheme_manual_002",
  "name": "现代轻奢软装方案 - 调整版"
}
```

行为：

- 仅允许源方案为 `confirmed`。
- 复制家具和风格配置。
- 不复制 render history。
- 新方案绑定相同户型版本，状态为 `draft`。
- 原方案保持不变。

`migrate` 请求：

```json
{
  "target_baseline_version_id": "v2",
  "id": "scheme_migrated_001",
  "name": "现代轻奢软装方案 - V2"
}
```

行为：

- 允许源方案来自历史版本。
- 目标必须是当前确认版本。
- 新方案状态为 `draft`。
- 不复制 render history。
- 执行房间 ID 映射、家具越界和悬挂引用校验。
- 无法安全映射的家具保留为 warning，不得静默删除。

### 16.3 写接口统一门禁

家具保存、AI 摆放、AI 效果图生成等所有写接口执行：

1. 读取方案。
2. 校验方案绑定的户型版本是当前 `confirmed`。
3. 校验方案状态为 `draft`；效果图生成可按产品需要允许 `confirmed`，但本批建议允许，以便定稿后继续生成成果。
4. 使用方案绑定的 geometry，不接受前端传入任意版本覆盖。

## 18. 迁移与回滚

### 18.1 迁移原则

- 迁移必须幂等，可重复运行。
- 迁移前执行项目目录和 artifacts 备份。
- 首次采用读兼容，不立即删除旧文件。
- 生产 D 项目与仓库样例数据可能不同，迁移以生产实际文件为准。

### 18.2 现有 D 项目迁移

1. 创建 `project.json`，当前版本指向 `v1`。
2. 复制根级 `geometry.json` 到 `baselines/v1/geometry.json`。
3. 创建 `baselines/v1/meta.json`，状态为 `confirmed`。
4. 为所有现有方案补充 `baseline_version_id=v1`。
5. 将 `default` 的显示名称迁移为“初始方案”。
6. 保留根级 geometry、furniture 和旧 artifacts 路径。
7. 校验默认方案 2 张、现代轻奢方案 1 张效果图仍可访问。

### 18.3 回滚

- 新版 API 上线前保留旧接口行为。
- 功能开关关闭时继续读取根级 geometry 和现有 scheme 结构。
- 回滚不得删除 `baselines/` 或改写历史成果。
- 部署健康检查除 `/api/health` 外，增加项目元数据和当前版本可读性检查。

## 19. 开发阶段与门禁

### 阶段 0：技术地基

- 建立 baseline 存储模块、项目级锁和迁移 dry-run。
- 补充数据备份、恢复和幂等测试。

完成门禁：

- 不改变现有生产读写行为。
- 全量既有 API/引擎测试通过。

### 阶段 1：户型版本后端

- 实现版本 CRUD、校验、确认和只读门禁。
- 根级 geometry 保持当前确认版本兼容镜像。

完成门禁：

- V1 迁移字节不改变原 geometry。
- 已确认版本保存返回明确冲突错误。
- V2 确认失败时当前版本指针不发生变化。

### 阶段 2：方案绑定与生命周期

- 方案绑定 baseline。
- 实现确认、调整副本、首选、归档和历史版本禁写。
- AI 和 render 全链路读取方案绑定版本。

完成门禁：

- 已确认方案无法原地修改。
- 调整操作产生新草稿且不复制成果。
- 历史版本方案无法新增成果。

### 阶段 3：前端信息架构

- 上线项目概览、户型基线、版本记录和统一上下文。
- 调整侧栏、面包屑和方案选择器。

完成门禁：

- 任意方案内页面均持续显示项目、版本、方案。
- 页面切换和刷新不丢方案上下文。

### 阶段 4：方案中心

- 当前/历史版本分组。
- 方案确认、调整副本、首选和归档交互。
- `default` 全面显示为“初始方案”。

完成门禁：

- 当前列表不混入历史方案。
- 所有破坏性操作有二次确认和明确目标。

### 阶段 5：方案对比

- 实现 2–3 套同版本方案选择和统一视图比较。
- 提供缺失成果空状态与跳转入口。

完成门禁：

- 不同版本不能进入同一对比。
- 切换视图时三列保持同一展示模式。

### 阶段 6：生产迁移与灰度

- 备份生产数据。
- 执行 dry-run 和正式幂等迁移。
- 先启用兼容读取，再启用版本写入，最后切换新版导航。

完成门禁：

- 生产健康检查通过。
- 现有家具和三张效果图完整可见。
- 旧接口在兼容期内仍可工作。

### 下一批：空房照片与实拍效果图

独立设计和实施：

- 户型版本级照片上传、房间和方向标注。
- 照片授权、删除与存储策略。
- 空房照片加方案轴测参考的多图生成。
- 实拍效果图按方案归档与比较。

本批不得提前耦合未确定的照片数据结构，只保留清晰扩展位。
