## 二、详细问题

### HIGH-01 家具加载失败可能导致已有数据被清空

- 严重程度：高
- 类型：功能性 Bug / 数据丢失
- 位置：`apps/web/src/components/studio/editor/hooks/useProjectData.ts:36`

#### 问题

`fetchFurniture()` 请求失败后，代码将家具状态和引用设置为空数组，并继续把整个项目标记为 `ready`：

```ts
try {
  const raw = await fetchFurniture(projectId);
  // ...
} catch {
  furnRef.current = [];
  setFurniture([]);
}
setLoadState('ready');
```

界面不会提示家具加载失败，家具编辑器仍然可用，保存操作会把当前空数组提交到后端。

#### 影响

网络抖动、接口超时或后端临时 500 均可能被误判为“项目没有家具”。用户点击保存后，已有家具数据可能被空数组覆盖。

#### 建议

- 家具加载失败应保留独立错误状态，不能等同于成功加载空数组。
- 加载失败期间应禁用家具保存。
- 应提供明确重试入口。

### HIGH-02 保存期间继续编辑会错误清除未保存状态

- 严重程度：高
- 类型：功能性 Bug / 数据丢失 / 异步竞态
- 位置：
  - `apps/web/src/components/studio/editor/hooks/useGeometryEditor.ts:198`
  - `apps/web/src/components/studio/editor/hooks/useFurnitureEditor.ts:707`

#### 问题

保存请求使用发起时的数据，但请求成功后无条件执行 `setDirty(false)`。保存按钮虽然在请求期间禁用，画布、侧栏表单和键盘操作仍然可继续修改数据，`Ctrl+S` 也可以直接再次调用保存函数。

典型时序：

1. 用户发起保存，请求提交版本 A。
2. 请求未结束时，用户继续编辑得到版本 B。
3. 版本 A 保存成功。
4. 前端执行 `setDirty(false)`，将版本 B 错误标记为已保存。

#### 影响

- 页面显示“已保存”，实际最新修改尚未写入后端。
- `beforeunload` 防丢失保护失效。
- 用户离开页面后丢失版本 B。
- 快速重复 `Ctrl+S` 还可能产生并发写入乱序。

#### 建议

- 保存时记录数据版本或引用，仅当当前数据仍等于已提交版本时清除 dirty。
- 保存期间禁用或串行化重复保存。
- 明确决定保存期间是否允许编辑；若允许，必须保留新修改的 dirty 状态。

---

### HIGH-03 根级 NoSSR 导致整个应用首屏依赖 JavaScript

- 严重程度：高
- 类型：首屏性能 / 可用性 / 架构
- 位置：`apps/web/src/app/AppWrappers.tsx:14`

#### 问题

根布局的所有子页面都被以下组件包裹：

```tsx
const NoSSR = dynamic(() => Promise.resolve(_NoSSR), {
  ssr: false,
});
```

生产构建生成的 `/studio/projects` HTML 中没有实际页面内容，而是包含 `BAILOUT_TO_CLIENT_SIDE_RENDERING`。这意味着静态生成的页面仍需等待客户端 JavaScript 加载和执行后才能显示。

#### 影响

- 增加首屏白屏时间和 LCP 风险。
- Next.js 静态预渲染与服务端渲染收益被抵消。
- 弱网和低性能设备体验明显下降。
- JavaScript 加载或执行失败时页面不可用。
- 搜索引擎和无脚本环境无法获得实际内容。

生产构建数据显示所有路由共享首屏 JavaScript 约为 103 kB，Studio 编辑器路由首屏约为 135 kB。

#### 建议

缩小客户端边界，仅将必须访问浏览器 API 或持有交互状态的 Provider、组件声明为客户端组件，不应在根布局关闭全部 SSR。

### MEDIUM-01 实时派生请求存在响应乱序

- 严重程度：中
- 类型：功能性 Bug / 异步竞态
- 位置：`apps/web/src/components/studio/editor/hooks/useGeometryEditor.ts:123`

#### 问题

`deriveSoon()` 使用 200ms debounce 减少请求频率，但已经发出的 `/derive` 请求不会被取消，也没有请求序号或输入版本校验。

如果旧请求耗时更长，可能出现：

1. 几何版本 A 发起请求。
2. 几何版本 B 发起请求。
3. B 响应先返回并显示正确结果。
4. A 响应后返回，将界面覆盖成旧结果。

#### 影响

派生墙、门窗和状态栏统计可能与当前几何数据不一致，直至用户再次编辑触发新的派生。

#### 建议

使用 `AbortController` 取消旧请求，或通过递增请求序号、输入版本标识丢弃过期响应。

### MEDIUM-02 D-only 静态部署仍展示其他项目

- 严重程度：中
- 类型：路由 Bug / 部署一致性
- 位置：
  - `apps/web/src/app/studio/projects/page.tsx:32`
  - `apps/web/src/app/studio/projects/[id]/layout.tsx:6`

#### 问题

`MVP_D_ONLY` 只隐藏了新建和删除操作，没有过滤项目接口返回的列表；动态路由静态生成却只枚举项目 `D`：

```ts
export function generateStaticParams() {
  return [{ id: 'D' }];
}
```

#### 影响

后端只要返回其他历史项目，项目卡片仍会显示。用户点击“打开”后会进入未导出的地址，在静态部署环境中返回 404。

#### 建议

D-only 模式下前端列表必须只显示 D，或在构建时生成所有可访问项目路由。部署模式与项目能力需要使用同一配置来源。

### MEDIUM-03 FullCalendar 与当前构建环境存在兼容问题

- 严重程度：中
- 类型：依赖兼容 / 潜在运行时 Bug
- 位置：`apps/web/src/components/calendar/EventCalendar.tsx:2`

#### 问题

生产构建报告：

```text
Attempted import error: 'unmountComponentAtNode' is not exported from
'react-dom' (imported as 'reactDom').
```

警告来源是 `@fullcalendar/react` 的旧版 VDOM 适配层。构建继续成功，但依赖使用了当前 React DOM 导出中不可用的 API。

#### 影响

`/admin/main/applications/calendar` 页面在挂载、卸载或路由切换时存在运行时异常风险。

#### 建议

升级 FullCalendar React 适配包至与当前 React/Next.js 兼容的版本，并对日历页面执行浏览器运行时验证。

### MEDIUM-05 触控取消事件没有清理捏合状态

- 严重程度：中
- 类型：移动端交互 Bug
- 位置：
  - `apps/web/src/components/studio/editor/hooks/useViewport.ts:190`
  - `apps/web/src/components/studio/ui/StageSvg.tsx:104`

#### 问题

双指缩放状态只在捕获阶段的 `pointerup` 中清理。`StageSvg` 没有向捕获层传递 `pointercancel` 处理函数。

系统手势、浏览器接管触摸、应用切换等情况均可能触发 `pointercancel`，导致 `pointers` Map 中残留无效触点。

#### 影响

后续双指缩放可能使用错误触点计算距离和中心，出现跳变、无法缩放等问题，严重时只能刷新恢复。

#### 建议

为 `pointercancel` 提供与 `pointerup` 等价的捕获层清理逻辑，并在组件卸载时清空触点状态。

### LOW-01 空格键会阻止整个编辑器页面滚动

- 严重程度：低
- 类型：交互 Bug
- 位置：`apps/web/src/components/studio/editor/hooks/useViewport.ts:82`

#### 问题

空格监听器挂载在 `window`，只排除了输入框等表单元素，没有检查焦点或指针是否位于画布。

#### 影响

编辑器页面中，即使用户正在侧栏或其他非画布区域操作，按空格也无法触发浏览器默认页面滚动。

#### 建议

仅在画布获得焦点或指针位于画布范围时启用空格平移，并让画布具有明确的可聚焦语义。

---

### LOW-02 确认弹窗缺少完整键盘和焦点管理

- 严重程度：低
- 类型：可访问性 / 键盘操作
- 位置：`apps/web/src/components/studio/ui/ConfirmDialog.tsx:95`

#### 问题

弹窗设置了 `role="dialog"` 和 `aria-modal="true"`，但没有：

- 打开后将焦点移入弹窗。
- 将焦点限制在弹窗内部。
- 支持 Escape 取消。
- 关闭后恢复触发按钮焦点。

#### 影响

键盘及屏幕阅读器用户可能无法可靠完成或取消删除确认流程。

#### 建议

采用具有完整焦点管理能力的 Dialog 组件，或补齐焦点圈定、Escape 和焦点恢复逻辑。

### LOW-03 Studio 核心功能缺少自动化测试

- 严重程度：低
- 类型：测试覆盖 / 回归风险
- 位置：`apps/web/src`

#### 问题

未发现前端测试文件。当前高风险业务路径均缺少自动化保护：

- 家具和几何加载失败。
- 保存期间继续编辑。
- 派生响应乱序。
- 草稿恢复与项目切换。
- Undo/redo 与拖拽提交。
- 静态导出动态路由。
- 多指手势与 `pointercancel`。

#### 影响

编辑器状态复杂且大量依赖异步请求、引用状态和全局事件，缺少测试会显著提高后续迭代的回归概率。

#### 建议

优先为数据丢失风险建立 Hook/集成测试，再补充编辑器关键用户路径的浏览器端测试。

## 四、建议修复优先级

### P0：数据安全

1. 修复家具加载失败后允许空数据保存的问题。
2. 修复保存请求与继续编辑之间的 dirty 状态竞态。
3. 防止重复保存请求乱序覆盖。

### P1：核心功能一致性

1. 为实时派生增加取消或过期响应保护。
2. 统一 D-only 项目列表与静态路由策略。
3. 验证并升级 FullCalendar 依赖。
4. 为上述路径补充自动化测试。

### P2：性能与体验

1. 移除根级 `NoSSR`，缩小客户端组件边界。
2. 补齐触控取消处理。
3. 限制空格平移快捷键作用范围。
4. 完善确认弹窗焦点管理。
5. 隐藏、禁用或实现当前无功能的公开控件。

## 五、审查边界与残余风险

本次审查以代码静态分析和构建验证为主，未覆盖以下内容：

- 未连接真实后端执行端到端业务回归。
- 未在桌面和移动浏览器中执行完整触控、拖拽和键盘测试。
- 未使用真实大户型、大量家具数据测量 INP、帧率和内存。
- 未执行 Lighthouse 或真实用户 Core Web Vitals 测量。
- 未验证生产 Nginx 静态导出环境下的全部路由行为。

因此，SVG 编辑器在大数据量下的交互性能、触控兼容性及真实部署路由仍存在需要运行时验证的残余风险。
