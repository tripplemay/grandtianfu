# 多候选家具方案实施规格

> 目标: 一个户型项目支持多套候选家具方案。用户可以从任一方案继续推进「家具精修 -> 轴测方案图 -> AI 照片级效果图 -> 空房实拍效果图」完整链路。本文为实施前规格,先定数据模型/API/前端路由/迁移策略,再写代码。

## 1. 术语

- `Project`: 户型项目。几何结构唯一,由 `geometry.json` 表示。
- `FurnitureScheme`: 家具候选方案。每套方案包含一份 `furniture.json`、方案元数据、该方案下的 AI render 历史。
- `default` 方案: 兼容旧数据的默认家具方案。旧的项目根级 `furniture.json` 等价于 `schemes/default/furniture.json`。
- `scheme_id`: 家具方案 id。路径安全规则沿用 project id: 仅 `[A-Za-z0-9_-]`。

核心不变量:

- 户型几何是项目级唯一真源: `{project}/geometry.json`。
- 家具是方案级真源: `{project}/schemes/{scheme_id}/furniture.json`。
- AI 摆家具只能创建新候选方案,不能直接覆盖现有方案。
- 画廊、照片级 AI render、未来空房实拍 render 都必须绑定 `scheme_id`。
- MVP 不引入服务端“当前方案”隐式状态; 页面用 URL query 显式携带 `scheme`。

## 2. 数据布局

当前布局:

```text
data/projects/D/
  geometry.json
  furniture.json
```

目标布局:

```text
data/projects/D/
  geometry.json
  furniture.json                    # 兼容旧端点; default 方案镜像或迁移源
  schemes/
    default/
      meta.json
      furniture.json
      renders.json
    scheme_ai_001/
      meta.json
      furniture.json
      renders.json
    scheme_ai_002/
      meta.json
      furniture.json
      renders.json
```

`meta.json`:

```json
{
  "id": "scheme_ai_001",
  "name": "现代轻奢 A",
  "source": "ai",
  "style_prompt": "现代轻奢,浅色石材,胡桃木",
  "base_scheme_id": "default",
  "status": "draft",
  "created_at": "2026-06-30T00:00:00Z",
  "updated_at": "2026-06-30T00:00:00Z"
}
```

字段约定:

- `source`: `legacy | manual | duplicate | ai`。
- `status`: MVP 只用 `draft | confirmed`; 不做复杂发布流。
- `base_scheme_id`: 复制或 AI 生成时的来源方案,用于追溯。
- `renders.json`: 只记录该方案的 AI 出图历史。旧根级 render history 可继续保留,但新写入都走方案级。

## 3. 兼容与迁移

### 3.1 读取兼容

如果项目没有 `schemes/`:

- `GET /schemes` 返回一个虚拟 `default` 方案。
- `default` 方案的 furniture 读取根级 `furniture.json`。
- 旧端点 `GET /furniture`、`POST /save-furniture` 仍读写根级 `furniture.json`。

### 3.2 首次写入迁移

第一次通过方案 API 写入时:

1. 创建 `schemes/default/`。
2. 从根级 `furniture.json` 复制到 `schemes/default/furniture.json`。
3. 写 `schemes/default/meta.json`。
4. 新的方案级保存写 `schemes/{scheme_id}/furniture.json`。

MVP 阶段,为了不破坏旧页面和旧接口:

- 保存 `default` 方案时同步写回根级 `furniture.json`。
- 非 `default` 方案不写根级 `furniture.json`。

## 4. 后端 API

旧 API 保留:

```text
GET  /api/projects/{house}/furniture
POST /api/projects/{house}/save-furniture
GET  /api/projects/{house}/render?mode=photo
POST /api/projects/{house}/render-ai
GET  /api/projects/{house}/renders
```

旧 API 默认操作 `default` 方案。

新增方案 API:

```text
GET  /api/projects/{house}/schemes
POST /api/projects/{house}/schemes
POST /api/projects/{house}/schemes/{scheme_id}/duplicate
GET  /api/projects/{house}/schemes/{scheme_id}
PATCH /api/projects/{house}/schemes/{scheme_id}
DELETE /api/projects/{house}/schemes/{scheme_id}

GET  /api/projects/{house}/schemes/{scheme_id}/furniture
POST /api/projects/{house}/schemes/{scheme_id}/save-furniture

GET  /api/projects/{house}/schemes/{scheme_id}/render?mode=plan2d|photo|shell
POST /api/projects/{house}/schemes/{scheme_id}/render-ai
GET  /api/projects/{house}/schemes/{scheme_id}/renders
```

删除规则:

- `default` 方案不可删除。
- 删除非 default 方案为软删: 移到 `{project}/schemes/.trash/{scheme_id}-{ts}`。
- 方案删除不删除 artifacts 文件; render history 不再从列表展示即可。后续清理脚本再做 GC。

创建方案请求:

```json
{
  "id": "scheme_manual_001",
  "name": "方案 A",
  "source": "manual",
  "base_scheme_id": "default",
  "furniture": []
}
```

复制方案请求:

```json
{
  "id": "scheme_copy_001",
  "name": "方案 A 副本"
}
```

返回摘要:

```json
{
  "id": "scheme_ai_001",
  "name": "现代轻奢 A",
  "source": "ai",
  "status": "draft",
  "items": 42,
  "renders": 3,
  "updated_at": "2026-06-30T00:00:00Z"
}
```

## 5. AI 摆家具接口

Phase 3 的 AI 摆家具不再写项目根级 `furniture.json`,而是生成候选 `FurnitureScheme`。

```text
POST /api/projects/{house}/furnish
```

请求:

```json
{
  "style_prompt": "现代轻奢,浅色石材,胡桃木,少量墨绿色点缀",
  "count": 3,
  "base_scheme_id": "default"
}
```

行为:

1. `room_brief.build_briefs(G)` 生成逐房简报。
2. LLM 只做选型: 每房从 `furniture_options` 中选择类型和数量。
3. 选型结果经受控校验: 未知类型、房间不允许类型、超出数量上限都拒绝或降级。
4. `floorplan_core/layout.py` 确定性落位。
5. `catalog.expand()` 补齐外观。
6. 每个候选写成独立 `FurnitureScheme`。
7. job result 返回候选方案摘要,不覆盖当前方案。

响应:

```json
{
  "job_id": "..."
}
```

job 完成结果:

```json
{
  "schemes": [
    {"id": "scheme_ai_001", "name": "现代轻奢 A"},
    {"id": "scheme_ai_002", "name": "现代轻奢 B"}
  ],
  "warnings": []
}
```

## 6. 前端路由与交互

保持当前静态导出路 A,不新增 `/projects/[id]/schemes/[scheme_id]` 动态段。用 query 传方案:

```text
/studio/projects/D/scheme
/studio/projects/D/editor?scheme=scheme_ai_001
/studio/projects/D/gallery?scheme=scheme_ai_001
/studio/projects/D/render?scheme=scheme_ai_001
```

页面职责:

- `/scheme`: 方案列表、方案复制/重命名/删除、AI 生成候选、选择某方案继续。
- `/editor?scheme=...`: 编辑该方案的家具。几何仍来自项目级 `geometry.json`。
- `/gallery?scheme=...`: 用该方案渲染 2D/photo/shell SVG。
- `/render?scheme=...`: 用该方案生成和查看 AI 照片级效果图。

MVP UI:

- 项目内页面标题区显示当前方案选择器。
- 如果 URL 没有 `scheme`,默认使用 `default`。
- 切换方案时只重载 furniture 和 render history,不重载 geometry。
- “AI 生成候选”按钮只在 `/scheme` 页出现。
- “应用/推进”不是复制覆盖,而是跳转到该方案的编辑器或 render 页。

## 7. artifacts 与 render history

新产物路径应包含 `scheme_id`:

```text
/api/artifacts/{house}/{scheme_id}/ai-render/{uuid}.png
```

旧产物路径保持可读,但新写入必须方案级隔离。

`renders.json` 从项目级迁移到方案级:

```text
schemes/{scheme_id}/renders.json
```

旧 `GET /api/projects/{house}/renders` 返回 `default` 方案历史。

## 8. 测试要求

后端:

- `list_schemes` 在无 `schemes/` 时合成 default。
- 首次方案写入会创建 `schemes/default` 且不破坏根级 `furniture.json`。
- `default` 保存同步根级 `furniture.json`。
- 非 default 保存不写根级 `furniture.json`。
- 删除 default 返回 400/409; 删除普通方案软删。
- render/render-ai 读取指定方案 furniture。
- `/furnish` 创建新方案,不覆盖任何已有方案。

前端:

- `studioApi` 支持 scheme API。
- `useProjectData(projectId, schemeId)` 能加载指定方案 furniture。
- 方案加载失败不能把远端家具解释为空数组。
- editor/gallery/render 的 URL 保持 `scheme` query。

引擎:

- `catalog.expand()` 幂等。
- `room_brief` 输出稳定。
- `layout.py` 输出确定性,并满足基本边界: furniture center 落在目标房间内。

## 9. 实施顺序

1. 后端方案存储层和 API,只做手工创建/复制/读写。
2. 前端接入 `scheme` query,让编辑器/画廊/render 支持指定方案。
3. render history 和 artifacts 按 scheme 隔离。
4. AI 摆家具生成 1..N 个候选方案。
5. `/scheme` 页做候选方案管理和生成入口。

每一步都必须保持旧 D 户型 `default` 方案可用。

## 10. 暂不做

- 不做跨方案 diff/merge。
- 不做服务端全局 active scheme。
- 不做多人协同或版本树。
- 不做自动删除 artifacts 的 GC。
- 不做方案级 geometry fork。几何分支是另一个大特性。
