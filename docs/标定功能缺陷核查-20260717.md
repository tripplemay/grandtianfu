# 透视标定功能全面核查 — UI 与逻辑缺陷清单（2026-07-17）

> 范围：`PerspectiveCalibrator.tsx`（输入采集 UI）→ `/calibration` 端点与校验（main.py）→ 求解器（`aigc/perspective.py`）→ 质量信号与门禁 → 存储与生命周期（baselines.py / calib_heal.py）→ 下游消费（第7步渲染）→ 测试覆盖。
> 方法：代码逐层审读 + 生产两案数据复现（798f23d3 / f4dab9bc）+ 合成数值实验（噪声鲁棒性 & 粗差可检测性）。
> 前置报告：`docs/AIGC链路核查-带评论效果图根因-20260717.md`（两张带评论效果图的根因定位）。

## 0. 总评

**求解器数学内核是健全的**：合成真值可精确复原（无噪声分毫不差）；诚实点击 + σ=8px 抖动下，2 锚点方案的家具盒偏移中位仅 ~34px、相机高度稳定（实验一）；calib-z-b1 修复的 z 翻转有回归测试锁住；binding/stale 生命周期设计正确（含 merge 兄弟房覆盖）。

**缺陷全部集中在"输入容错"设计上**，可概括为三句话：

1. **UI 容易诱发粗差**（对应关系整个弄错，而非点不准）：虚拟角点、罗盘方位心算、角标自动分配、写死的墙向文案；
2. **2 锚点 + 每组恰 2 条线的输入方案让粗差不可检**：冗余度≈0，某些粗差能被求解器完美拟合（实验二 case A：角标互换 → 自报误差 0.0px、物理指标全部正常、家具盒偏移 651px——**原理上不存在能拦住它的残差门禁**）；
3. **算出来的质量信号被全链路丢弃**：reprojection_error 只存盘，UI 不显示、保存不拦、渲染不查；相机高度 sanity 的概念存在于迁移工具里，从未上线。

## 1. 缺陷清单

严重度定义：CRIT=已在生产造成用户可见错图 / 结构性不可检；HIGH=大概率造成错图或放行坏输入；MED=显著增加出错概率或阻碍恢复；LOW=打磨项。

## 2. 实验证据

**实验一（诚实点击 + 高斯抖动，300 trials/格）**：合成真值相机（斜视 30°、俯 8.5°、f=1450、高 1400mm），对全部 12 个点击坐标加噪。σ=8px（约对应模态里 ±3 屏幕像素）下：2 锚点盒偏移中位 34px / P90 65px，相机高中位 1404mm；3、4 锚点仅微幅改善。**结论：点击抖动不是事故来源；数学内核对诚实输入足够稳。**

**实验二（用户级粗差注入）**：

| 注入 | 结果 |
|---|---|
| 两锚点角标互换（SW↔SE） | **自报 0.0px，相机高 1400mm，盒偏移 651px —— 完全静默** |
| 单锚点角标错选（SW 标成 NW） | 自报 631px（可检测——若有门禁；今日生产会无声存盘） |
| 线组画反（x↔y） | 物理门当场拒绝 ✓ |
| 3 锚点其一角标错 | 物理门当场拒绝 ✓（**≥3 锚点把静默灾难变成当场报错**） |
| 3 锚点全对 | 精确复原 ✓ |

**生产两案对照**：798 = A3+B1 型粗差（线画错）+ D1 放行，自报 2353px；f4d = A1+G1 型粗差（虚拟角猜点）+ D1 放行，自报 112px。

## 3. 修复建议（按缺陷 ID 映射）

核心判断：**预览（人验）、≥3 锚点（检测力）、门禁（执行）三件缺一不可**——case A 证明纯阈值方案有原理性盲区；纯 3 锚点产生的信号没人读；纯预览依赖用户认真看。三者合围后每类粗差至少被两道防线覆盖。

**P0（止损，1 批次内）**
1. 保存 + 渲染双端质量门（D1/D3/B2）：reprojection_error > ~50px 拒绝（阈值依据：诚实点击 P90≈23px，留 2 倍余量；对角线比例 ~2.5%）；
2. 物理门收紧（C2）：相机高 ∉ [800, 2200]mm 或相机位置离绑定房（merge 组并集外扩 1m）过远 → 拒绝；顺手用 C3 焦距范围（HFOV 35-110°）做第三道廉价 sanity；
3. 输入语义校验（B1）：线长 ≥ 图幅 5%、组内两线夹角/交点合理、"水平边"斜率上限、锚点距图缘 ≥2%、锚点像素/世界坐标去重；
4. **标定完成即预览**（A4）：用响应里的 camera 把房间线框（地脚线+顶角线+2.7m 墙高竖棱）反投叠加到照片，同屏显示误差数值与评级；"确认无误"后才落盘（两步提交）；
5. 删除标定端点 + UI 入口（B3）。

**P1（治本，随后批次）**
6. 锚点 ≥3 强制（C1，即 `BL-calib-min-3-anchors`）+ 去掉角标自动分配、改为点击时高亮平面图对应角（A2）；
7. 锚点源重做（A1/A8/G1）：merge 组取成员并集包络角点；允许选任意可见实体特征（门框/窗角，世界坐标由几何 openings 派生）；同名房间在绑定 UI 显示区分徽标与迷你平面图高亮；
8. 文案动态化（A3）：按房间几何生成"沿 X 向的两条水平边（如 XX 墙上下沿）"并配示意图；
9. direction 交叉校验（D2）：解算 yaw 与 v0-v3 象限反向即拒；
10. UI 支持每组 >2 条线（A6）。

**P2（打磨）**
11. 标定历史/回看编辑（A5/E1）；heal 重算误差字段（E2）；缩放放大镜（A7）；负样本测试补齐（F1-F3，随 P0 门禁一起写）。

## 4. 根治方案：从"输入抽象对应关系"到"所见即所得对齐"（2026-07-17 追加）

§3 是对现有交互的加固；本节回答"是否存在更好的范式，彻底修复"。答案：有。现有设计的病根是**要求用户生产抽象对应关系**（罗盘角标、方向线分组）——人对这类任务天然易错（19 项缺陷里 A1/A2/A3/A9/G1 全是它的衍生物），且输入方案零冗余使错误不可检（C1）。根治 = 让对应关系**由构造保证正确**，用户只做"对齐"这一件视觉任务。

### 4.1 方案空间

| 路线 | 内容 | 评价 |
|---|---|---|
| **一（推荐主干）** | **特征点 PnP + 实时线框对齐**：系统按 room+direction 预置初始相机，把房间 3D 线框（地脚线/墙棱/门窗开口）叠加到照片；用户从线框上**拖特征点**到照片对应位置，每落一点即重解位姿并重绘线框；残差+物理门通过才允许保存 | 对应关系错误被构造消灭（点从模型上拿，自带世界坐标，不存在"标错角"）；画线步骤整个砍掉；预览即人工验收；≥4 点天然冗余 |
| 二（增强层） | **VLM 预标定**：用已有 chat_json 基建（semantic_accept 模式）让 VLM 按几何先验（该房该方位应见哪扇窗/门）输出特征点初始像素 → 预摆好，用户只微调确认 | 把标定从"操作"降为"确认"；单次调用成本远低于一张错图；VLM 像素精度不足由用户微调兜底，失败降级为纯手动 |
| 三（保底） | 现有"线+角"模式收编为**专家模式**：§3 P0 的动态文案+门禁全套 | 特征点稀缺的极端照片（纯白墙）用；实现在 §3 已列 |
| 四（不推荐现在做） | 全自动 CV 消失点（引入 cv2 依赖+鲁棒性长尾）、深度学习单图标定/房间布局估计（体量与栈不符）、多照片重建（超出单照产品形态） | 投入产出不成比例；路线一+二已把人工成本压到"拖 4 个点" |

### 4.2 路线一关键设计

1. **可以砍掉画线步骤的数学依据**：≥4 个特征点对（世界坐标已知）→ 标准 PnP。焦距未知时用"焦距扫描（HFOV 40-110°，~40 档）× 每档 DLT/EPnP × 取最小残差 + Gauss-Newton 精化"，纯 numpy ~百行，确定性、可单测。现有 `calibrate()` 的 anchors 参数**本来就接受任意 (world,px) 且不限 z=0**（`_solve_poses` 全 3D 求解）——"只能点房间矩形角"从来只是 UI 限制。`calibrate()` 保留为专家模式求解器，新增 `solve_pnp()` 并行存在，产物同为 Camera dict，存储/渲染零改动。
2. **特征点来源（按可靠性排序）**：门框竖边×地面交点（openings 的 axis/at/span 给出精确 (x,y,0)，25 个开口全可用）＞实体墙角（含 merge 组内墙）＞落地窗框角（南窗 sill=0）＞普通窗台角（**数据缺口：openings 无 sill/header 高度字段**——补数据或该特征附高度微调滑杆）。
3. **merge 组**：线框用 `axon.slice_geom_for_room` 的整组几何（已有函数），特征点池 = 全组成员的墙角+开口；f4d 的"虚拟门厅角"问题自然消失。
4. **⚠ 两个 z 世界红线**：线框墙高必须用 perspective 的 `_REAL_CEILING_MM=2700`，**严禁**借 axon/scene 的 1450（decor-envelope-b1 的教训原样适用于线框渲染）。
5. **交互闭环**：初始位姿 =（房间中心偏 direction 反向、高 1400mm、HFOV 70°）；拖点落定 → 解算（后端 dry-run 端点或 TS 端口二选一，前者单一真源，防抖 ~200ms 足够）→ 线框重绘；未过门禁（残差>50px / 相机高 ∉[800,2200] / 出房 / HFOV 出界）保存按钮禁用并显示原因。**保存的 payload 即点对列表，向后兼容现有 anchors 结构**；binding 指纹扩展纳入所引用 openings。
6. **降级路径**：特征点 <4 或用户要求 → 专家模式（线+角，带 §3 门禁）。两模式产物同构，渲染端无感。

### 4.3 分期落地（每期独立可交付）

| 期 | 内容 | 体量 | 消灭的缺陷 |
|---|---|---|---|
| **A（小时级速赢）** | 现有模态提交前加 **dry-run 预览**：后端 `?dry_run=1`（跑 `_calibration_camera`+误差，不落盘，~10 行）；前端拿返回 camera 投影房间 8 角画线框叠加 + 显示误差评级，"确认无误"才真提交 | 0.5-1 天 | A4、D1 的 UI 半边 |
| **B（1 个批次）** | §3 P0 全套硬门禁（保存+渲染+物理+输入语义）+ 删除标定端点 | 小 | B1/B2/B3/C2/D1/D3 |
| **C（1-2 个批次）** | 特征点 PnP 模式（solve_pnp + 特征点池派生 + 拖拽对齐 UI + 门禁联动），线+角转专家模式 | 中 | A1/A2/A3/A6/A8/A9/C1/G1（结构性消灭） |
| **D（1 个批次，可选）** | VLM 预标定初始化 + 窗 sill 高度数据补齐 | 小-中 | 把标定成本降为"确认" |

依赖关系：A、B 无依赖可先行；C 依赖 B 的门禁框架；D 依赖 C 的特征点池。
