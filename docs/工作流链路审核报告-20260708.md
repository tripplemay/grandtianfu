# 工作流链路审核报告

- 日期: 2026-07-08
- 范围: 从创建项目到最终生成实拍效果图的完整产品工作流
- 角色: 工作流审核专家
- 目标: 判断当前链路设计是否有助于生成准确、符合用户需求的实拍效果图, 并提出修改方案
- 约束: 本报告只做审核分析与建议, 不修改业务代码

## 一句话结论

当前项目的工程链路已经比较完整: 项目、户型基线、方案、家具、空房照片、轴测效果图、实拍效果图之间有清晰的数据边界, 后端也有较多防误用保护。它已经能支持生成较准确的实拍效果图。

但从产品工作流角度看, 当前流程更像一条工程管线, 还不是一条完整的“用户需求到设计交付”流程。真正影响实拍图是否符合用户需求的几个关键环节, 例如用户设计 Brief、照片质量门禁、轴测确认、实拍图反馈迭代, 尚未成为主流程中的强约束节点。

因此, 下一阶段优先应优化工作流, 而不是先换模型或大改底层生成调用。

## 当前主链路

当前可识别的主链路如下:

1. 项目台打开项目。
2. 户型基线页查看或编辑户型版本。
3. 确认户型基线。
4. 方案中心创建手动方案或 AI 风格候选。
5. 家具编辑器调整方案家具。
6. 轴测效果图页生成 `axon-photoreal` 效果图。
7. 户型基线页上传并标注空房照片。
8. 实拍效果图页选择空房照并生成 `real-photo` 实拍图。
9. 在方案中心选择首选方案, 或在效果图页下载产物。

这个流程的数据方向是正确的: 先锁定空间, 再锁定布局和风格, 最后用真实照片作为结构锚点生成实拍图。

## 逐步审核

### 1. 项目创建

当前状态:

- 后端支持 `POST /api/projects` 创建项目。
- 前端项目台当前处于 `MVP_D_ONLY = true`, 新建项目入口被隐藏。
- 后端新建项目默认生成一个最小合法单房间起步几何。

合理性判断:

- 对 D 户型 MVP 和工程验证是合理的。
- 对真实业务不够合理。真实实拍图准确性的第一前提是真实户型几何。如果项目创建只能从一个默认单房间开始, 用户需要大量手工重建户型, 误差会沿着后续链路持续放大。

对实拍效果图的影响:

- 若户型几何不真实, 后续家具落位、轴测参考和实拍匹配都会偏。
- 即使 image2 能保留照片结构, 第二张轴测参考也会给模型错误的家具相对位置。

修改方案:

- 将项目创建改成向导式入口:
  - 选择新建方式: 导入户型图、上传底图描摹、复制已有项目、从模板开始。
  - 上传户型底图后先标定比例。
  - 描摹墙体、门窗、窗面。
  - 命名房间。
  - 保存为户型草稿版本。
- 如果继续保留最小单房间 starter, 应明确标记为“测试/空白模板”, 不应作为真实项目默认路径。

### 2. 户型基线

当前状态:

- 户型版本有 draft / confirmed / superseded 状态。
- 草稿可编辑, confirmed 版本作为方案共同空间基础, 历史版本只读。
- 确认户型后进入方案中心。
- 空房照片绑定到 baseline version, 不绑定方案。

合理性判断:

- 这是当前链路中最合理的设计之一。
- 户型基线作为所有方案和照片的空间真源, 能避免后续方案和效果图被几何变更污染。

对实拍效果图的帮助:

- 方案和实拍照片都绑定同一 baseline version, 能确保轴测参考和真实照片属于同一户型版本。
- 历史版本只读能保证旧图可追溯。

不足:

- “户型确认”和“空房照片管理”目前都在基线页, 但没有形成实拍生成准备度检查。
- 用户可以确认户型, 但并不一定完成照片、房间、视角等 AIGC 必备信息。

修改方案:

- 在基线页增加“实拍生成准备度”面板:
  - 是否有空房照片。
  - 是否每张照片标注 `room_id`。
  - 是否选择 `direction`。
  - 是否所有照片 `purpose=empty`。
  - 是否有墙面材质参考图。
  - 是否存在房间窗面/门洞缺失。
- 将“确认户型”与“准备实拍素材”拆成两个明确里程碑。

### 3. 空房照片管理

当前状态:

- 上传图片会归一化为 JPEG, 物化 EXIF 方向, 剥离 GPS, 压边, 记录尺寸和 sha256。
- 照片支持 `room_id`、`direction`、`note`、`purpose`。
- 实拍页当前已过滤 `purpose == empty/null` 的照片。
- 后端 `render-real` 当前也已对非 empty 照片做硬校验。

合理性判断:

- 图片安全和隐私处理是合理的。
- 照片绑定 baseline version 也是合理的。
- `room_id` 和 `direction` 对实拍图准确性非常关键, 当前已有选择器和视角预览, 方向正确。

不足:

- `room_id` 和 `direction` 仍然不是强制门槛。未标注房间时仍允许走整宅参考, 只是提示质量较差。
- 照片质量本身没有评分或门禁, 例如过暗、过曝、严重广角畸变、遮挡、水印、非空房照片等。
- 空房照片上传默认未显式写 `purpose=empty`, 通过 null 兼容旧数据。长期看不够清晰。

对实拍效果图的影响:

- 未标注房间时, 轴测参考会退回整宅, 目标房间占比小, 模型更容易串房间家具。
- 未选拍摄方向时, 家具靠墙关系容易错。
- 照片质量差时, image2 很可能优先复制低质量结构, 出图上限受限。

修改方案:

- 将 `room_id` 和 `direction` 设为实拍生成前的默认必填项。
- 如果允许未标注照片生成, 应改成显式“低准确度模式”, 并要求用户确认。
- 上传空房照时显式写 `purpose=empty`。
- 增加照片质量检查:
  - 分辨率和比例。
  - 是否为空房室内照。
  - 是否包含强水印/人物/遮挡。
  - 亮度、清晰度、畸变。
- 在实拍页展示“照片可用性评分”。

### 4. 方案创建与 AI 候选

当前状态:

- 方案绑定当前 confirmed baseline。
- 手动方案未带家具时, 会从当前基线家具拷贝标准布局。
- AI 方案生成基于已有方案, AI 不重新落位, 只生成风格候选和可选同组换件。
- 方案 `style_prompt` 可后续编辑。

合理性判断:

- “AI 不重新落位”是正确的。它减少了家具漂移、房间错配和非法坐标。
- 方案绑定 baseline 能保证空间基础一致。
- `style_prompt` 可编辑是必要能力, 有助于迭代用户需求。

不足:

- 对全新项目来说, 如果基线没有可靠标准家具布局, AI 候选缺少好的布局基底。
- 用户需求仍主要是自由文本和少量风格预设, 缺少结构化 Brief。
- AI 候选的评价标准不明确: 用户不知道应该比较布局、风格、材质还是实拍可行性。

对实拍效果图的影响:

- 如果基础布局质量不高, 实拍图也只会把不合理布局渲染得更真实。
- 如果风格输入不完整, image2 会自行补全, 结果可能不符合用户偏好。

修改方案:

- 将方案创建拆为两个里程碑:
  - 标准布局方案: 先完成家具数量、位置、房间用途。
  - 风格候选方案: 再生成不同材质、色彩、软装风格。
- 新增结构化设计 Brief:
  - 居住人群。
  - 预算档位。
  - 风格方向。
  - 主材和禁用材质。
  - 主色和禁用颜色。
  - 保留硬装 / 可替换软装。
  - 重点房间。
  - 不希望出现的元素。
- AI 候选应明确展示“布局相同, 风格不同”, 避免用户以为 AI 在重新设计平面。

### 5. 家具编辑器

当前状态:

- 方案模式编辑方案家具。
- baseline 草稿模式编辑基线家具。
- 保存前后有家具校验和场景校验。
- 历史/归档/未知方案只读。

合理性判断:

- 编辑权限设计较合理。
- 家具数据与场景校验能在出图前暴露风险。

不足:

- 从用户工作流看, “标准布局是否已完成”没有明确完成动作。
- 家具编辑器更多是工程编辑器, 对非专业用户缺少“实拍生成影响”的提醒。

对实拍效果图的影响:

- 家具是否贴墙、是否挡门、是否与窗冲突, 会直接影响实拍图可信度。
- 用户如果只看到 2D/轴测编辑器, 未必意识到这些问题会影响 image2。

修改方案:

- 增加“布局准备完成”状态, 通过后才能进入 AI 风格候选。
- 家具保存后给出更面向出图的诊断:
  - 大件是否靠墙。
  - 沙发/床头是否靠窗。
  - 是否挡门。
  - 是否有房间缺少核心家具。
  - 是否有客餐厅动线拥堵。

### 6. 轴测效果图生成

当前状态:

- `render-ai` 先做场景校验。
- 校验通过后生成 photo 轴测底图, 再调用 image2。
- prompt 中已贯通 `style_prompt`。
- provider 实际返回尺寸也已记录为 `actual_size`。

合理性判断:

- 轴测图作为实拍图之前的设计预检是合理的。
- 它能帮助用户先判断布局、风格和整体质感, 再决定是否消耗实拍图额度。

不足:

- 主工作流引导当前只把“生成效果图”作为里程碑, 没有区分“轴测效果图确认”和“实拍效果图确认”。
- 用户可以不确认轴测就直接去实拍页。

对实拍效果图的影响:

- 如果轴测参考本身风格或家具不满意, 实拍图会继承这些问题。
- 没有轴测确认会让实拍图生成变成盲试。

修改方案:

- 增加“确认轴测方案”步骤:
  - 确认家具布局。
  - 确认风格材质。
  - 确认重点房间。
  - 确认没有重大场景警告。
- 只有确认后的方案才作为推荐实拍生成对象。
- 如果允许跳过确认, 应提示“未确认轴测, 实拍结果可能偏离预期”。

### 7. 实拍效果图生成

当前状态:

- 实拍页要求显式携带 scheme。
- 只显示 empty/null 目的照片。
- 选中照片后可选择/保存拍摄方向。
- 后端按照片 room_id 切片轴测参考, 按 direction 旋转。
- 后端 prompt 已贯通方案 `style_prompt`。
- record 保存 `style_snapshot`、`requested_size`、`actual_size`、`photo_sha256`、`wall_photo_ids`。

合理性判断:

- 这条链路当前已经比较合理。
- 空房照作为结构锚点, 轴测图作为方案锚点, 风格 prompt 作为软装约束, 三者组合能帮助 image2 输出更准确的实拍效果图。

不足:

- 生成前用户看不到完整输入栈: 空房照、轴测参考图、风格快照、墙面材质参考图、请求尺寸。
- 没有强制照片标注和方向选择。
- 没有实拍图质量反馈闭环。

对实拍效果图的影响:

- 用户可能误选照片或误选视角。
- 用户不知道当前实拍图会继承哪套风格 prompt。
- 出图不满意后, 用户不知道应该改照片、改家具、改风格还是改墙面材质。

修改方案:

- 在生成前增加确认面板:
  - 选中空房照。
  - 对应 room_id 和房间名。
  - 当前 direction 和四视角预览。
  - 本次使用的轴测参考图。
  - 方案 `style_snapshot`。
  - 墙面材质参考图列表。
  - 请求尺寸。
- 默认强制 `room_id` 和 `direction` 已填写。
- 增加生成后反馈入口:
  - 结构错。
  - 家具位置错。
  - 风格不符。
  - 材质不符。
  - 画质问题。
  - 其他。
- 根据反馈跳转到对应修正点。

### 8. 方案首选与交付

当前状态:

- 方案中心可以设置首选方案。
- 轴测效果图页生成后可设为首选。
- 实拍页提供下载和删除。

合理性判断:

- 首选方案作为工作流收口是合理的。
- 下载和历史保留也是必要能力。

不足:

- “首选方案”和“最终实拍图验收”不是同一个概念。当前流程容易把方案首选当成最终交付完成。
- 实拍页没有“设为最终交付图 / 通过验收”的动作。

修改方案:

- 增加实拍图级别的交付状态:
  - draft / accepted / rejected。
  - accepted 的实拍图可以作为项目最终封面或交付图。
- 项目概览应展示“最终实拍图”而不只是最近效果图。

## 当前链路中已经做得好的点

1. Baseline 版本化清晰, 能保护历史结果。
2. Scheme 绑定 baseline, 避免跨版本误生成。
3. AI 软装方案不重新落位, 只做风格候选, 降低漂移风险。
4. 上传图像做了格式验证、EXIF 方向物化、隐私剥离和压缩。
5. 实拍图生成已按 room_id 切片, 按 direction 旋转轴测参考。
6. 实拍照片用途前后端都已过滤/硬校验。
7. 第 7 步实拍 prompt 已贯通 `style_prompt`。
8. 出图记录已保存 prompt、usage、style snapshot、actual size 等可追溯信息。

## 主要问题清单

### P0: 主工作流没有覆盖“空房照片”和“实拍图验收”

当前项目概览的工作流引导主要覆盖:

- 确认户型
- 创建方案
- 布置家具
- 生成效果图
- 选定首选

但最终目标是实拍效果图, 主流程却没有把“上传/标注空房照”和“生成/验收实拍图”纳入核心 stepper。

修改建议:

- 将工作流引导扩展为:
  - 确认户型
  - 完成标准布局
  - 创建/选择方案
  - 确认轴测效果
  - 上传并标注空房照
  - 生成实拍效果图
  - 验收最终图

### P0: 实拍生成缺少强准备度门禁

当前未标注房间或未选视角仍可生成, 只是降低质量。

修改建议:

- 默认要求 `room_id` 和 `direction` 必填。
- 如果用户跳过, 必须显式确认低准确度模式。

### P1: 用户需求采集过于自由文本化

自由文本适合快速试错, 但不适合稳定满足用户需求。

修改建议:

- 增加结构化设计 Brief。
- 由 Brief 编译为第 5 步和第 7 步不同 prompt。
- 保留自由文本作为补充, 不作为唯一输入。

### P1: 轴测图缺少确认动作

轴测图是实拍图前最好的预检产物, 但当前没有“确认轴测方案”的流程门。

修改建议:

- 生成轴测图后要求用户选择一张作为当前方案参考。
- 实拍图默认使用已确认轴测图。

### P1: 生成后缺少反馈闭环

用户看到不满意结果后, 目前只能手工猜该改哪里。

修改建议:

- 增加不满意原因选择。
- 将原因映射到具体修正入口:
  - 结构错 -> 照片/房间标注。
  - 家具位置错 -> direction 或家具编辑。
  - 风格不符 -> style_prompt / Brief。
  - 材质不符 -> 墙面材质参考。

### P2: 项目创建不适合真实多项目生产

D-only 部署对 MVP 合理, 但不适合完整产品。

修改建议:

- 多项目模式开启前, 必须补齐导入户型图、标定比例、描摹和初始基线确认向导。

## 推荐目标工作流

建议把最终工作流设计为 8 个阶段:

1. 创建项目
   - 输入项目名、户型名、客户需求。
   - 选择导入方式。

2. 建立户型基线
   - 导入户型底图。
   - 标定比例。
   - 描摹墙体、门窗、房间。
   - 校验通过后确认 baseline。

3. 建立标准布局
   - 基于 baseline 放置核心家具。
   - 校验门窗、动线、靠墙、大件位置。
   - 标记布局完成。

4. 填写设计 Brief
   - 风格、预算、主材、主色、禁忌、重点房间。
   - 可上传参考图。

5. 生成/选择方案
   - AI 基于标准布局生成风格候选。
   - 用户比较、编辑、选择方案。

6. 确认轴测效果
   - 生成轴测效果图。
   - 用户确认一张作为实拍参考。

7. 准备实拍素材
   - 上传空房照。
   - 标注房间和视角。
   - 校验照片质量。

8. 生成并验收实拍效果图
   - 展示生成前输入确认面板。
   - 调 image2 生成。
   - 用户验收或反馈重试。
   - 选定最终交付图。

## 优先级建议

### 第一优先级

- 扩展项目概览工作流 stepper, 纳入空房照片和实拍图验收。
- 实拍生成前增加 Readiness Gate。
- 默认强制 `room_id` 和 `direction`。

### 第二优先级

- 增加结构化设计 Brief。
- 增加轴测确认动作。
- 增加生成前输入确认面板。

### 第三优先级

- 增加实拍图反馈闭环。
- 增加照片质量评分。
- 增加最终交付图状态。

## 最终判断

当前工作流能支撑“技术上跑通实拍图生成”, 也已经具备较好的工程安全性和可追溯性。

但如果目标是稳定生成“准确且符合用户需求”的实拍效果图, 当前流程还需要从工程链路升级为交付链路。核心改造方向不是增加更多按钮, 而是把用户需求、照片质量、轴测确认和结果反馈变成主流程中的必经节点。

只要补齐这些工作流门禁, 现有 image2 链路的质量会更稳定, 也更容易把每次不满意结果转化为可执行的修正动作。
