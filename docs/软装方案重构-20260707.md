# 软装方案中心重构（2026-07-07 定案）

## 背景与核心判断
方案中心原设计里"AI 生成新方案时自动摆放家具"实际不符合常理。**根因不是 LLM 能力限制,而是设计:** `furnish.py` 里 LLM 被明确禁止摆放(system prompt「不要输出坐标」),只选"房间/类型/数量";真正的摆放是 `layout.plan_report` 的确定性网格填充算法(26 个固定分数网格点 + 贪心避门),零设计语义。要做好自动布局需从头造布局引擎(高风险大投入),故**下线自动落位**。

## 定案(用户确认)
1. **家具下沉到基线,人工锁定**:户型+家具都在户型基线里由人手工编辑,确认即锁定。
2. **方案 = 从基线拷贝的副本**(决策 1-B 拷贝模型,非差分):方案仍存完整 `furniture.json`,渲染路径不变;编辑器默认"换件/改尺寸、不挪位"。
3. **换件换尺寸**:锚点保持、footprint 可变,场景校验器标重叠,方案编辑器给微调(决策 2)。
4. **墙面材质共享**:留在基线几何,方案不覆盖(决策 3)。
5. **AI 重生 = 风格驱动选件**(不摆位):给定风格 → 为每槽位从 swap-group 选件 + 生成 style_prompt → N 个风格候选(同一布局的 N 种风格)。目录先加 swap-group+风格标签元数据编现有 46 类;**风格变体基元扩充留后续**(决策 4:先跑通主流程再扩家具库)。
6. **迁移**:基线 v1 家具 = 现在初始方案(default)的家具 = 根 `furniture.json`(决策 5)。

## 分阶段路线
- **A · 家具下沉基线**（本文件对应,PR 进行中）:基线存/拷/锁家具 + D 迁移 v1 furniture + 基线草稿编辑器家具可写。渲染/golden 不变,方案照旧。
- **B · 方案层改造** ✅上线:建方案从基线家具拷种子(create_scheme manual 空布局→read_baseline_furniture);换件=swapFurnitureType(改类型保中心+采新尺寸+保 id/orient/rot/zorder/label/color);方案编辑器默认锁位(posLocked=!baselineVersionId, gate 拖/缩放/旋转/nudge/对齐分布, 🔒/🔓 切换)。3-agent 审查修 nudge/align 绕锁 + swap 丢标签。
- **C · AI 重生 + 目录编组**:目录 swap-group+风格标签;furnish 改风格选件;`layout.plan_report` 自动落位下线;"生成候选"= 同布局 N 风格。
- **D · 方案中心 + 工作流收尾**:生命周期简化(**砍掉 scheme 级 confirm** —— 已确认全链路无操作要求 confirmed,它只把方案锁死逼 fork 增殖)、单一"+新建"、实拍页全链路接入、升级 v2 空态"从上一版迁移"CTA、双数据源收敛、参数命名统一。
