# 阅天府 Studio 画布能力差距分析与对齐路线(最终综合报告)

> 基于「画布核心交互 / 几何编辑深度 / 家具对象编辑 / 性能反馈移动端」4 视角审查综合而成。本报告仅分析,不改动任何代码。所有结论已以代码为准核实(grep + 全文阅读),关键利好:坐标换算用 `getScreenCTM().inverse()` + `meta.origin`,与未来视口变换天然兼容;纯函数沉淀在 `lib/floorplan`、写入口收敛于 `updateG`/`updateFurniture` 两处——这使大量增强是「加层」而非「重构」。

## ② 对标主流的差距清单(按 P1/P2/P3)

### P1 — 核心缺失,体验硬伤

**P1-1 视口缩放/平移(pan/zoom/滚轮/Fit)** · severity HIGH
- 现状:viewBox 静态取 `meta.canvas_viewbox`,`StageSvg` 等比塞容器且 `touch-none` 禁原生捏合。整张 D 户型被缩到 ~0.3–0.4 倍,无法放大精修门窗/家具,无平移、无 Fit/100%/缩放到选区。
- 主流(Figma/Konva/酷家乐):滚轮以光标为锚缩放、空格/中键拖平移、捏合、Fit/100%/Zoom-to-selection、百分比显示。
- 落地:新增 `useViewport` 持 `{scale,tx,ty}`,在 `StageSvg` 内包一个被 `transform` 的 `<g>`(或动态改 viewBox);绑 `onWheel`(ctrl=缩放/否则平移)+空格平移+双指。**因 CTM inverse 已正确反算,所有现有 hit-test/拖拽逻辑零改坐标**。
- 改动面/复杂度:中。新增 hook + 改 `StageSvg.tsx` + 两 Stage 透传;`touch-none` 配合手势保留。
- 需引入库:否(纯 SVG transform 足够)。

**P1-2 undo/redo 历史栈(几何+家具共栈)** · severity HIGH
- 现状:`updateG`/`updateFurniture` 直接替换 state + 镜像 ref,无历史。误拖/误删/误 merge-split 不可逆,只能 reload 丢未存改动。
- 主流:统一 Ctrl+Z/Y,拖拽**落点入栈一次**(非每帧)。
- 落地:在写入口集中挂 `past/present/future` 快照栈(G 不大,快照式即可);**关键**:pointermove 期不入栈,仅 `onSvgPointerUp`/侧栏 commit 时 push 一帧;几何+家具共栈(否则 Tab 切换后 undo 语义混乱)。
- 改动面/复杂度:中,收益极高。结构性改动之一。
- 需引入库:可选(use-undo 等)或自研。

**P1-3 键盘交互(Delete/Esc/方向键/Ctrl-S)** · severity HIGH(Delete/Esc)·MED(方向键)
- 现状:全仓零 `onKeyDown`。删除必点侧栏 🗑;插入模式只能再点 toggle 退出;无方向键微移、无 Ctrl 组合。
- 主流:Delete 删选中、Esc 取消插入/清选、方向键 1 单位 +Shift 10 单位、Ctrl+Z/C/V/S。
- 落地:顶层 `FloorplanEditor` 加全局 keydown 按 mode 分发,复用现成 `onDel*`/`computeMove`/`reanchor`;输入框聚焦需 `e.target` 守卫防误删。
- 改动面/复杂度:小(主要接线)。
- 需引入库:否。

**P1-4 拖拽实时尺寸标注 / 吸附辅助线** · severity HIGH
- 现状:`bestSnap/snapEdge` 静默吸附无可视;拖动/缩放过程画布上无 W×H、无对齐红线,尺寸只在侧栏数字。与主流体感差距最大的一项。
- 主流:Figma/Sketch/酷家乐拖动即显对齐红线 + 实时尺寸/间距 tooltip。
- 落地:`computeMove/Resize` 额外返回命中候选边 → 临时 `snapGuide`/`hud` state → Stage 加一层覆盖渲染(参考现有 `fwPts` 临时层)。
- 改动面/复杂度:中。`geometry.ts` 扩返回 + 新增 Guide/Dimension 覆盖层。
- 需引入库:否。

**P1-5 移动端不可放大(touch-none 无缩放兜底)** · severity HIGH
- 现状:`touch-none`(pointer 拖拽必需)+ 无 pan/zoom = 手机上整张户型缩到屏宽极小,无法放大查看/操作。
- 主流:户型类移动端必备双指捏合 + 单指平移。
- 落地:**依赖 P1-1**;视口落地后接管多指 pointer(两指 pinch/pan,单指拖元素)。
- 改动面/复杂度:中(随 P1-1)。
- 需引入库:否。

**P1-6 防数据丢失(beforeunload + dirty 标志)** · severity HIGH
- 现状:grep 无 `beforeunload`,纯手动保存,无 dirty 守卫;编辑后刷新/切路由全丢且无提示。几何/家具两个独立保存按钮,易只存一边。
- 主流:有未保存改动时拦截卸载并提示。
- 落地:写入口置脏、保存成功清脏;`window.beforeunload` 脏时拦截;保存按钮显示 ●未保存/✓已保存。
- 改动面/复杂度:小,收益高。
- 需引入库:否。

**P1-7 画布无法新增/删除房间** · severity HIGH(几何)
- 现状:工具栏只有 ＋门/＋自由墙/打通/分隔,`useGeometryForm` 无 `onAddRoom/onDelRoom`;新建户型/删冗余房只能改底层 JSON。
- 主流:Planner5D/酷家乐拉框建房 + 选中删除 + 右键菜单。
- 落地:`useGeometryForm` 加 `onAddRoom/onDelRoom`;画布加「＋房间」两点拉矩形插入模式(类比 freewall);Delete 复用。
- 改动面/复杂度:中。1 hook + 1 工具按钮 + 1 canvas insert 分支。
- 需引入库:否。

### P2 — 重要

**P2-1 拖拽期 per-event setState 触发整树重渲染(无 memo)** · severity MED-HIGH
- 现状:每个 pointermove 都 `updateG→setG`/`setFurniture(map)` 整数组重建;所有层无一个 `React.memo`(grep `memo(`=0)。拖一房=重渲全部房+墙+门窗+只读家具层。当前规模体感尚可,但叠加 derive 后是放大规模的硬天花板。
- 主流:Konva/tldraw 拖拽期 imperative 直改被拖元素,松手才 commit;子项普遍 memo。
- 落地:①给 `RoomRect/FurnitureItem/WallLine/OpeningMarker/DerivedWallsLayer` 加 `React.memo`(props 已基本原始值);②拖拽期只动被拖元素(局部 state 或改 `<g>` transform),pointerUp 再落库+derive。
- 改动面/复杂度:中(集中在两 canvas hook + item 组件)。需引入库:否。

**P2-2 家具自由旋转(画布旋转柄 + 角度吸附)** · severity MED
- 现状:`Orient='N'|'S'|'W'|'E'`,无角度字段,渲染无 rotation;斜放餐椅/转角沙发无法表达。
- 主流:任意角度旋转 + 顶部旋转柄 + 15°/45° 吸附(Shift 自由)。
- 落地:数据加 `rot:number`(度),外层 `<g transform="rotate(rot,cx,cy)">`;命中/反推仍用中心,`reanchor` 不改;加旋转柄 + 侧栏角度输入。**注意同步轴测/导出管线对 orient 的依赖**。
- 改动面/复杂度:中。需引入库:否。

**P2-3 家具画布缩放手柄** · severity MED
- 现状:家具尺寸只能侧栏键入,画布只有 move;与房间 8 把手体验不一致。
- 主流:选中即出包围盒 + 角/边手柄拉伸。
- 落地:**直接复用** geometry 的 `ResizeHandles` 模式,新增 `furnResizeRef` 分支,改 w/h(圆形改 r)。
- 改动面/复杂度:中(~40 行 + 新组件)。需引入库:否。

**P2-4 复制/粘贴/复制副本(Ctrl+C/V/D、Alt+拖)** · severity MED
- 现状:无 clipboard/duplicate;家具只能从默认模板新建,配好的件无法复制。
- 主流:Ctrl+C/V、Ctrl+D 偏移副本、Alt+拖出副本。
- 落地:`duplicateSelected` 深拷贝+偏移 push;剪贴板 state 支持跨房粘贴。与 P1-3 键盘、P2-7 多选一起做最自然。家具尤其简单。
- 改动面/复杂度:小-中。需引入库:否。

**P2-5 家具吸附 + 越界/碰撞约束** · severity MED-HIGH
- 现状:`reanchor` 仅取整,无网格/贴墙/家具间吸附、无碰撞;未命中房间仍更新 dx/dy(保留旧 room_id),家具可飘到墙外,语义混乱。
- 主流:Magicplan/酷家乐贴墙自动吸附+朝墙背、家具间边缘吸附、越界/碰撞高亮。
- 落地:分阶段 — (a)越界 clamp/红描边提示(低成本先做);(b)近墙贴墙吸附+按边设 orient;(c)家具间包围盒吸附(高成本)。新增 `clampToRoom`/`snapToWall` 纯函数(可单测)。
- 改动面/复杂度:中。需引入库:否。

**P2-6 命中区域过小 + 无 hover + 触控目标不达标** · severity MED
- 现状:把手 12u(≈4px 屏幕)、自由墙线 2u(≈0.7px)、门窗选中后反而变细(10→6),均无透明 hit-padding;无 hover 态;拖拽中 cursor 不切 `grabbing`;远低于 44px 触控建议。
- 主流:不可见 hit-line(transparent 粗 stroke `pointer-events:stroke`)、hover 高亮、把手随 zoom 反比保持恒定屏幕尺寸。
- 落地:线状元素叠透明宽命中线;把手 `width={12/scale}`;加 `onPointerEnter/Leave` hover;选中态只加粗可见线、命中域恒定。
- 改动面/复杂度:小(各层加命中层)。需引入库:否。

**P2-7 多选 / 框选(marquee)/ 全选** · severity MED
- 现状:几何 room2 仅服务打通,家具 selFurn 单值;无 Shift 加选、无拖框、无 Ctrl+A、无群组移动/批量删。
- 主流:Shift 加选、空白拖 marquee、Ctrl+A、组合移动、对齐分布。
- 落地:selection 升级为 id 数组/index 集合(结构性改动之二);空白起拖画 marquee → up 按包含/相交判定。建议排在 P1 + id 化之后。
- 改动面/复杂度:大。需引入库:否。

**P2-8 门窗画布交互薄(无端点拖宽/翻转)** · severity MED
- 现状:`OpeningMarker` 只整体沿墙滑(不改宽);改宽/翻 hinge/swing/door_type 全靠侧栏;选中态线宽反而变细难抓。
- 主流:酷家乐门窗两端把手拖宽 + 画布翻转箭头。
- 落地:加两端 resize handle(复用 host clamp)+ 翻转小按钮;修选中线宽。
- 改动面/复杂度:中。需引入库:否。

**P2-9 自由墙不可拖动(仅选中)** · severity MED
- 现状:派生墙 `pointerEvents:none`(仅开门模式可点);自由墙有 onPointerDown 但仅 setSelection,改位置只能侧栏改 at/span。
- 主流:直接抓墙拖移是户型基操。
- 落地:FreeWall 加 move 拖拽(复用 computeMove 思路平移 at/span + 吸附);派生墙拖动需映射回房间边、复杂度高,自由墙优先。
- 改动面/复杂度:中。需引入库:否。

**P2-10 家具/对象稳定 id(去 index 身份)** · severity MED(正确性)
- 现状:`FurnitureLayer key={i}`,`selFurn` 为下标,`onDelFurn` 按下标 filter;删中间件后下标前移,React 复用错节点,undo/重排会选错件。`Date.now()%100000` 生成 id 有撞号风险。
- 主流:每对象稳定 uid,key/selection 都用 id。
- 落地:`buildDefaultFurniture` 生成 id;key/selFurn 改 id;统一 id 生成器(计数器/nanoid)。**是 P2-4/P2-7/undo 的前置基础,建议优先补**。
- 改动面/复杂度:小-中。需引入库:可选(nanoid)。

**P2-11 axis 用自由文本框无校验** · severity MED(违反边界校验规则)
- 现状:opening/freewall 的 `axis` 用 `<TextRow>` 接受任意字符串,输错(如 "x")直接破坏渲染分支。
- 落地:改 `SelectRow options={['h','v']}`。一行改动,低成本高收益。需引入库:否。

**P2-12 画布定位校验反馈不足 + 家具保存零校验** · severity MED / LOW-MED
- 现状:画布高亮只有重叠房红框;conflicts/warns/门窗越界只在侧栏文字,无法点击定位;家具 `onSaveFurn` 直落盘,可存出界/重叠。
- 落地:conflict 文案带元素 id,点击→选中高亮;家具保存前校验中心是否落房内 + 警告。
- 改动面/复杂度:中 / 低。需引入库:否。

**P2-13 z-order / 叠放控制(类型已有 z 但未用)** · severity MED(家具)
- 现状:`z?` 字段存在但 `FurnitureLayer` 按数组 index 渲染,无置顶/置底;rug 与其上家具层序靠数组顺序碰运气。
- 落地:渲染前 `sort((a,b)=>(a.z??0)-(b.z??0))`(需解耦显示序与数据序,保留 origIndex);侧栏加置顶/置底;rug 默认 z=-1。
- 改动面/复杂度:小-中。需引入库:否。

### 产品边界类(需显式声明,非简单补缺)

- **多边形/L 形/斜墙房间**:`Room.rect:[x,y,w,h]` 全基于 AABB,`buildFreeWall` 强制 h/v。L 形只能两矩形+打通拼,斜墙不支持。升级为 `polygon` 需重写 derive/overlap,属大改里程碑(见 ⑤)。
- **墙厚不可调**:线宽是渲染粗细非物理墙厚,无字段。若引擎 derive 已有墙厚则暴露到侧栏,否则属引擎层缺口。

## ③ 对齐主流最该先补的 Top 8(含依赖顺序)

依赖关系是排序核心:**视口变换是众多能力的前置,稳定 id 是多选/复制/undo 的前置,历史栈是误操作兜底的前置。**

1. **稳定 id 化(P2-10)** — 小而关键的地基。先做,解锁 P2-4/P2-7/undo,消除 index 错位隐患。
2. **视口缩放/平移(P1-1)** — 收益最大、坐标层零改。是 P1-5 移动端、P2-6 把手恒定尺寸、精修一切操作的前置。
3. **键盘 Delete/Esc/方向键(P1-3)** — 小改动补基本手感;为 Ctrl+Z/C/V 铺好监听层。
4. **undo/redo 共栈(P1-2)** — 落点入栈;误操作硬伤兜底,越早做越省(写入口集中)。
5. **防丢失 beforeunload + dirty(P1-6)** — 极小改动,防"刷新全丢"的灾难性体验。
6. **拖拽实时尺寸 + 吸附辅助线(P1-4)** — 与主流体感差距最大,户型工具的核心可视反馈。
7. **命中区加宽 + hover + 拖拽期 memo(P2-6 + P2-1)** — 一组体验/性能改善,随视口落地后顺势做(把手随 scale 反比)。
8. **复制副本 + 家具旋转/缩放手柄(P2-4 + P2-2 + P2-3)** — 功能补齐,依赖键盘层与稳定 id;家具手柄可直接复用 `ResizeHandles`。

> 之后再进入多选/框选(P2-7)、家具吸附贴墙(P2-5)、画布增删房间(P1-7)。其中 P1-7 虽属 P1 但工程上依赖键盘删除与 insert 模式框架,故置于此批。移动端捏合(P1-5)随第 2 步视口自然附带。

## ④ 架构建议:现方案能撑到哪 + 分阶段路线

### 受控 inline SVG 现方案的天花板
- **能撑到**:当前单户型(几十~百级元素)+ 单选 + 静态视口。底层三块做得对——CTM inverse 坐标换算、纯函数+不可变、相对坐标家具模型(room_id+dx/dy)——使得 zoom/pan、旋转、手柄、吸附线等增强多为「加层」。
- **何时引入视口变换层**:立刻(Top 2)。在 `StageSvg` 内包一个 transform `<g>` 或受控 viewBox 即可,**无需换技术栈**,因 CTM inverse 已兼容。
- **何时引入历史栈**:在多对象/复杂操作前(Top 4)。挂在 `updateG`/`updateFurniture` 两个唯一写入口,快照式即可(G 不大)。
- **拖拽性能临界点**:当「per-event setState 整树重渲」遇到「元素破百 + derive 叠加」时(P2-1)。先上 `React.memo` + 拖拽期 imperative 直改被拖元素,可显著延后换栈需求。
- **何时考虑换 Konva/canvas**:仅当单图元素破千、或需高频动画/海量家具阵列时。**当前不必换**——瓶颈是 per-event setState 不是 SVG 本身;且换 Konva 会损失现有 inline SVG 的可访问性/调试/导出便利。先把 memo + 松手提交做到位再评估。

### 分阶段对齐路线

**阶段 0 — 地基(1-2 项,低风险)**
范围:稳定 id 化(P2-10)、axis 下拉(P2-11)、pointercancel(P3)、方向光标(P3)。
价值:解锁后续多项,消除正确性隐患。风险:极低。

**阶段 1 — 视口与导航(高价值,低坐标风险)**
范围:`useViewport`(zoom/pan/Fit,P1-1)+ 移动端捏合(P1-5)+ 把手随 scale 恒定(P2-6 部分)。
价值:即刻可用性飞跃,移动端解锁。风险:中(手势与 touch-none/pointer-capture 协调需测;CTM 兼容已验证降低风险)。

**阶段 2 — 安全网与手感(小改动,高回报)**
范围:undo/redo 共栈(P1-2)+ 键盘层(P1-3)+ beforeunload/dirty(P1-6)+ 复制副本(P2-4)。
价值:误操作可逆 + 不丢数据 + 基本快捷键,补齐"编辑器标配"。风险:低-中(历史栈是结构性改动,但入口集中)。

**阶段 3 — 可视反馈与性能(体感对齐主流)**
范围:实时尺寸 + 吸附辅助线(P1-4)+ memo/松手提交(P2-1)+ 命中加宽/hover(P2-6)+ 网格/标尺(P3)。
价值:与 Figma/酷家乐的"拖动即见对齐线+尺寸"对齐;放大规模天花板。风险:中。

**阶段 4 — 对象编辑深化**
范围:家具自由旋转(P2-2)+ 缩放手柄(P2-3)+ 家具吸附/越界(P2-5)+ z-order(P2-13)+ 门窗端点拖拽/翻转(P2-8)+ 自由墙拖动(P2-9)+ 画布增删房间(P1-7)。
价值:画布操作与侧栏对齐,减少"拖完再去侧栏改"两步。风险:中(旋转需同步轴测/导出管线对 orient 的依赖)。

**阶段 5 — 多选与库**
范围:多选/框选/对齐分布(P2-7)+ 家具库重构(分类/搜索/缩略图/拖入,P3)+ 定位校验反馈(P2-12)+ 无障碍(P3)+ 自动保存草稿(P3)。
价值:批量布置效率 + 专业感。风险:较大(selection 单选→数组是结构性重构,建议最后)。

## ⑤ 明确「不必做 / 暂不做」(防过度)

- **换 Konva/canvas/WebGL**:当前规模不需要。先做 memo + 拖拽 imperative;仅元素破千或需高频动画再评估。
- **多边形/L 形/斜墙房间**:取决于产品定位。若定位"标准矩形户型 + 打通拼 L 形",则**暂不做**;需在 UI/文档显式标注"矩形房 + 合并"的产品约束,避免用户误以为是 bug。若未来要做,单列里程碑(rect→polygon 重写 derive/overlap,大改)。
- **家具顶点级/自由变形**:色块阶段无意义,等接入真实素材再议。
- **家具翻转 flip**:色块阶段镜像无视觉差异,暂不做;接入不对称真实图形后随旋转一起做。
- **等距/智能分布吸附**:Figma 级锦上添花,优先级最低,可后置或不做。
- **复杂自动保存/协同**:本工具是单人离线编辑,localStorage 草稿足矣,无需实时协同/版本树。
- **墙厚物理建模**:若引擎层无墙厚数据,不应在前端硬造;标注为引擎缺口待支持,前端暂不暴露伪墙厚。
- **派生墙直接拖动**:映射回房间边复杂度高、收益有限;自由墙拖动优先,派生墙暂不做。
