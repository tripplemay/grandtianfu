# AIGC 核心链路审查与整改建议

日期：2026-07-11

范围：本次审查聚焦核心 AIGC 链路：创建户型基线 -> 生成方案 -> 生成轴侧效果图 -> 生成实拍效果图。目标是判断当前设计是否足以让用户稳定生成准确适配户型、效果满意的效果图，并给出后续开发 agent 可执行的整改方向。

## 结论

当前工程已经具备一个可运行的结构化 AIGC 闭环：用户维护基线户型和家具，系统生成方案候选，再基于结构化场景生成轴侧图和实拍图。该架构方向是成立的，也能在样例项目中生成视觉上可用的结果。

但它还不足以稳定承诺“准确适配户型、效果满意”。主要原因是：户型和家具真值依赖人工且质量门槛偏软；方案生成并不生成布局；轴侧图确认没有真正成为实拍图的输入约束；实拍 geometry-lock 虽然提升了落位能力，但标定、投影、验收和用户可见质量反馈仍不够硬。

后续整改建议优先围绕三件事推进：

1. 把户型/家具/照片/标定的基线质量做成硬门槛。
2. 明确“轴侧 -> 实拍”的产品语义，并在代码中真正串联或明确解耦。
3. 把实拍生成的几何锁定、自动验收和质量反馈从软提示升级为可追踪、可失败、可复现的质量系统。

## 当前链路复盘

### 1. 创建户型基线

新项目初始化时会生成一个默认矩形房间，基线版本由 `baselines` 模块管理。基线确认主要依赖几何校验，允许用户在当前 confirmed baseline 上创建新 draft。

关键代码：

- `apps/api/main.py:239`：`_starter_geometry` 创建默认单矩形房间。
- `apps/api/main.py:378`：创建项目时写入初始 `geometry.json`、空 `furniture.json`，并初始化 baseline。
- `apps/api/baselines.py:570`：新项目从 `v1` draft 开始，没有天然 confirmed baseline。
- `apps/api/baselines.py:744`：保存 geometry 时调用 `geometry.validate`。
- `apps/api/baselines.py:828`：confirm baseline 时主要检查 geometry ERROR。
- `packages/floorplan_core/floorplan_core/geometry.py:723`：几何校验覆盖面积、重叠、开口等，但不校验家具和照片标定一致性。

风险判断：

- 默认基线不是用户真实户型，只是一个编辑起点。
- 几何校验偏结构合法性，不等于真实户型准确性。
- baseline confirm 不强制家具、照片、标定同时达到可渲染质量。
- 新 baseline 会复制旧照片和旧标定，几何变化后可能复用过期标定。

### 2. 生成方案

当前方案生成是“基于已布置家具的风格候选”，不是“从户型自动生成可用家具布局”。AI 只输出风格提示和可选同组家具替换，不输出坐标和尺寸。

关键代码：

- `apps/api/furnish.py:1`：模块注释明确说明 AI 不摆放家具，只做风格服务。
- `apps/api/furnish.py:62`：prompt 要求不要输出坐标、尺寸和解释。
- `apps/api/furnish.py:138`：应用 swap 时保留中心点，只替换同组家具。
- `apps/api/main.py:1367`：`/furnish` 提交异步方案生成任务。
- `apps/api/schemes.py:403`：创建 scheme 时绑定 baseline，必要时复制当前 baseline 家具。

风险判断：

- 如果用户没有先人工完成准确家具布局，AI 不会补齐布局能力。
- 异步生成过程中，如果 current baseline 变化，存在旧布局绑定到新 baseline 的风险。
- swap 是类型/房间维度的替换，缺少更细粒度的实例级控制。
- 生成候选没有在创建时做完整 scene 级预校验，问题可能延迟到渲染阶段暴露。

### 3. 生成轴侧效果图

轴侧 AI 图的输入来自结构化 scene：先渲染程序化 axon SVG/PNG，再交给图像模型编辑。prompt 会强调保持相机角度、几何、墙体、门窗和家具位置。

关键代码：

- `apps/api/main.py:1510`：`_render_ai_response` 生成轴侧 AI 图。
- `packages/floorplan_core/floorplan_core/scene.py:492`：`build_scene` 生成可渲染 scene。
- `packages/floorplan_core/floorplan_core/scene.py:747`：`validate_scene` 做场景校验。
- `packages/floorplan_core/floorplan_core/prompt_gen.py:92`：生成轴侧图提示词。

风险判断：

- 轴侧图有输入校验，但没有对 AI 输出做几何回检。
- 图像模型可能改变墙体、门窗、家具比例和细节。
- 部分提示词包含硬编码环境描述，例如固定朝向/窗型倾向，可能与真实户型不一致。
- scene 中存在自动调整家具尺寸/高度/位置的逻辑，实际渲染可能与用户编辑的原始布局不完全一致。

### 4. 生成实拍效果图

实拍生成有两条路径：

- geometry-lock：有相机标定时，基于空房照片和家具投影彩色盒子生成。
- fallback：没有可用标定时，基于空房照片和程序化轴侧参考生成。

关键代码：

- `apps/api/main.py:1991`：`_render_real_geometry_lock` 处理 geometry-lock 实拍。
- `apps/api/main.py:2182`：`_render_real_response` 处理实拍主入口和 fallback。
- `apps/api/main.py:2054`：geometry-lock 输入为空房照片 + guide 图。
- `apps/api/main.py:2272`：fallback 会重新渲染程序化 axon 作为参考。
- `apps/api/aigc/perspective.py:88`：基于消失点和地面锚点做相机标定。
- `apps/api/aigc/perspective.py:248`：生成彩色投影盒子 guide。
- `apps/api/aigc/acceptance.py:136`：自动验收检查。

风险判断：

- 已确认轴侧图不会作为实拍图的输入，实拍图不能继承用户确认过的轴侧视觉结果。
- geometry-lock guide 主要表达盒子和类型，不表达实例级材质、款式、SKU、参考图和软装细节。
- 标定可选且可能过期；没有强绑定 geometry hash、room hash、image size、重投影误差。
- 自动验收偏启发式，失败后仍可能选 best attempt 返回，前端没有充分展示失败原因。

## 关键问题清单

### P0-1：基线质量门槛不足

问题：

当前 baseline confirm 主要检查 geometry ERROR，不足以证明这个基线可用于高质量 AIGC。家具布局、照片 room/direction、相机标定、标定与当前几何的一致性都没有作为硬门槛。

影响：

用户可能在错误户型、空家具、错误照片标定上继续生成方案和效果图，后续 AIGC 再强也只能生成“看起来合理但不适配真实户型”的图。

建议修改：

- 增加 baseline readiness API，返回可生成质量状态，而不是只返回 geometry validation。
- confirmed baseline 至少应校验：
  - 有有效房间和外轮廓。
  - 门窗/开口挂载合法。
  - 家具非空或明确标记为“只做空房/硬装”。
  - 家具在房间内，旋转后不穿墙，不遮挡门口核心通道。
  - 空房照片必须绑定 `room_id`、`direction`、`purpose=empty`。
  - 标定必须绑定当前 `geometry_hash`、`room_rect_hash`、`image_size`，并有可接受的重投影误差。
- geometry 或 room metadata 变化后，相关照片标定自动置为 stale。

验收标准：

- 用户不能在 readiness 为 blocking 的 baseline 上启动方案生成、轴侧生成或实拍生成。
- 前端明确展示 blocking/warning 项和修复入口。
- 后端所有生成入口都再次校验 readiness，不能只依赖前端。

### P0-2：方案生成没有锁定提交时的 baseline 快照

问题：

`/furnish` 提交时会检查 base scheme 是否属于当前 baseline，但异步任务真正创建新 scheme 时，如果 current baseline 已变化，可能把旧布局绑定到新 baseline。

影响：

方案与户型版本错配，后续轴侧/实拍生成会出现不可解释的偏差。

建议修改：

- 提交 furnish job 时记录 `baseline_version_id`、`geometry_hash`、`furniture_hash`、`base_scheme_id`。
- 异步任务创建 scheme 时显式传入提交时的 `baseline_version_id`。
- 如果当前 baseline 已变化，应根据产品策略选择：
  - 直接失败并提示用户重新生成。
  - 或创建历史 baseline scheme，但标记只读，不允许继续生成新效果图。

验收标准：

- 异步任务完成时不会隐式绑定“完成时的 current baseline”。
- 所有生成记录都能追溯到提交时的 baseline 和 scheme 快照。

### P0-3：轴侧确认没有约束实拍生成

问题：

实拍生成没有使用用户确认过的轴侧 AI 图。前端也有注释说明确认轴侧“不改实拍的 img2img 输入”。这会导致用户认为“我已经确认轴侧满意，实拍应该沿用它”，但实际并不成立。

影响：

轴侧和实拍可能在家具造型、材质、软装细节上漂移，用户满意度不可控。

建议修改：

需要先做产品决策：

- 方案 A：真正实现“轴侧 -> 实拍”。实拍生成输入加入 accepted axon image，或从 accepted axon 中提取结构化风格/材质/家具参考，再进入 real render prompt。
- 方案 B：明确解耦。UI 文案和流程改成“轴侧预览”和“实拍独立生成”，不要把轴侧确认表达成实拍前置质量保证。

更推荐方案 A，因为用户自然会把轴侧确认理解为最终实拍的一部分约束。

验收标准：

- 若要求 has_confirmed_axon，则实拍生成记录必须保存 `accepted_axon_render_id`。
- provider 输入或 prompt 中必须包含 accepted axon 的可验证引用。
- 实拍 UI 展示“本次实拍基于哪张轴侧图”。

### P0-4：实拍自动验收是软门槛且用户不可见

问题：

geometry-lock 路径会做自动验收和 retry，但最终可能返回 `auto_check.ok=false` 的 best attempt。前端没有充分展示 auto_check、失败原因、retry 次数和低置信程度。

影响：

系统知道图可能不合格，但用户看到的是一张普通成功结果，无法判断是模型失误、标定问题还是布局问题。

建议修改：

- API 返回并持久化完整 auto_check 摘要：`ok`、`score`、`fail_reasons`、`attempts`、`selected_attempt`、`geometry_lock_enabled`。
- 前端在 real-render 结果卡片展示质量状态：
  - geometry-lock 已启用/未启用。
  - auto_check 通过/失败/跳过。
  - retry 次数。
  - 低置信原因。
- 对 P0 质量问题默认不自动标记为可交付结果，至少需要用户显式确认。

验收标准：

- auto_check 失败的结果在 UI 上不能与普通成功结果长得一样。
- 失败原因可被用户或开发 agent 直接定位。
- retry 使用的真实 prompt、输入图、provider response、usage 都可追踪。

### P0-5：照片标定生命周期缺失

问题：

baseline 新版本会复制旧照片和标定；更新照片 room/direction 时不会清空标定；设置标定时没有记录几何版本哈希和误差指标。

影响：

geometry-lock 可能在错误房间、错误尺寸、错误视角上投影家具，生成“落位看似锁定、实际错位”的实拍图。

建议修改：

- 标定数据增加：
  - `geometry_hash`
  - `room_rect_hash`
  - `room_id`
  - `image_width`
  - `image_height`
  - `reprojection_error`
  - `created_from_baseline_version_id`
- 修改 geometry、room rect、photo room_id、photo direction、image asset 后，标定自动 stale。
- real render 启动时只接受 fresh calibration；stale calibration 必须 fallback 或阻断。

验收标准：

- 任意会影响投影的编辑都会让旧标定失效。
- 前端能展示“需重新标定”的明确状态。

## P1 改进项

### P1-1：补齐布局生成能力或强化人工布局流程

如果产品希望普通用户从户型自动得到完整方案，就需要引入布局生成/优化能力。可以先从规则模板做起，而不是直接交给 LLM：

- 按房间类型生成候选家具组合。
- 用几何约束检查门窗、通道、墙距、碰撞。
- 生成多个布局候选并打分。
- AI 只负责风格和偏好，不直接产出不可控坐标。

如果短期不做自动布局，则 UI 和流程应强制用户先完成家具布置，并把“AI 不移动家具”作为生成前检查项，而不是只在说明文案中提示。

### P1-2：增强 scene validation

建议补齐：

- 家具旋转后的真实 footprint 校验。
- 家具与家具碰撞校验。
- 门口/窗前净空从 warning 升级为可配置 blocking。
- raw furniture 与 axon-adjusted furniture 的差异报告。
- ceiling height、cutaway wall height、家具物理高度分离，避免渲染高度和真实投影高度混用。

### P1-3：增强 geometry-lock guide 表达能力

当前彩色盒子能帮助模型落位，但对“满意效果”不够。建议 guide/prompt 加入：

- 每个家具实例的 label。
- 材质、颜色、风格关键词。
- SKU 或 catalog reference。
- 软装、灯具、挂画、地毯等非盒状元素的独立引导层。
- 已确认轴侧图作为风格和造型参考。

### P1-4：实拍照片采集质量门槛

实拍效果强依赖空房照片质量。建议新增照片 readiness：

- 分辨率和宽高比检查。
- 是否有人、遮挡、水印、强反光。
- 是否覆盖目标墙面/地面。
- 是否严重广角畸变。
- 同一房间是否有足够视角覆盖。

这些检查可以先从规则和人工标记开始，再逐步加入 VLM。

### P1-5：生成记录可复现

每次 AIGC 生成记录应保存：

- provider、model、size、seed 或等价参数。
- 所有输入图片 asset id/hash。
- 实际 prompt，包括 retry hint 后的版本。
- scheme/baseline/geometry/furniture/style/brief 快照 hash。
- 每次 attempt 的 usage 和错误。

这样开发 agent 才能稳定复现和定位漂移。

## P2 改进项

### P2-1：户型导入能力

目前 schema 偏轴对齐矩形和水平/垂直墙体。真实户型常见斜墙、异形空间、柱、梁、飘窗、非正交墙。建议后续支持：

- CAD/PDF/图片导入。
- 多点标尺和自动矢量化。
- 非正交墙体。
- 柱、梁、窗台、吊顶等结构元素。

### P2-2：建立 AIGC 回归评测集

建议建立真实样本集，覆盖不同房间、户型、视角、光照和照片质量。每次改 prompt、provider、geometry-lock 或 validation 后都跑评测：

- 几何适配评分。
- 家具落位评分。
- 材质/风格一致性评分。
- 明显失败类型统计。
- 人工可接受率。

## 建议执行顺序

1. 先修 P0-2 baseline 快照绑定，避免继续产生版本错配数据。
2. 实施 P0-5 标定生命周期，防止 geometry-lock 使用过期投影。
3. 增加 P0-1 baseline readiness，并把三个生成入口接入后端硬校验。
4. 做 P0-4 auto_check 前后端透出，让系统已知失败对用户和开发者可见。
5. 决策并实现 P0-3：实拍是否真正依赖 accepted axon。
6. 再推进 P1 scene validation、layout generation 和 guide 表达能力。
7. 最后扩展导入能力和长期回归评测集。

## 最低验收矩阵

| 场景 | 期望结果 |
| --- | --- |
| 空家具 baseline 启动方案生成 | 阻断或明确进入“仅硬装/空房”模式 |
| baseline 几何修改后使用旧照片标定 | 标定 stale，不能进入 geometry-lock |
| furnish 异步期间切换 current baseline | 新 scheme 不会绑定到错误 baseline |
| 没有 accepted axon 但启动实拍 | UI/API 根据产品策略阻断或明确提示实拍独立生成 |
| geometry-lock auto_check 失败 | UI 明确展示失败和原因，不伪装成普通成功 |
| provider retry 后选择第二次结果 | 记录真实 selected attempt、真实 prompt、真实 usage |
| 家具旋转后穿墙 | scene validation 能识别并阻断或高危提示 |
