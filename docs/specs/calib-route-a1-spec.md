# calib-route-a1 — 从图像本身取几何（消失点 / Manhattan 约束）

> 状态：**待用户 plan 批准（spec lock）**。分支 `feat/calib-route-a1`（off b3 HEAD）→ PR → squash；绝不单推 `main`。
> 前置：`calib-cure-b3` 已收口（`docs/test-reports/calib-cure-b3-signoff-2026-07-21.md`）。
> 用户裁决（2026-07-21）：路线调整为 A。

## 1. 为什么换路线

b3 收口签收 §3 的三个结构性病根，一句话概括：

> **我们把 7 个自由度里最难的一步（像素 ↔ 抽象平面角的对应）交给了人，而人做错时系统测不出来。**

决定性数据：生产 13 份存档标定中，走新特征点模式的是 **0**；用户 4 轮真实浏览器 L2 **4 轮全败**；隔离验收实测正对单墙位形中 **26.3%** 解出「在门内但错」的相机而 reproj 接近 0。

**A 路线的核心主张：旋转与焦距应当从图像本身取，而不是从人的对应取。**

室内照片天然满足 Manhattan 假设（墙、地、天花三组互相正交）。三组正交消失点可直接给出旋转 `R` 与焦距 `f` —— 那是 7 个自由度里的 4 个，且**不依赖任何人工对应**。剩下 3 个（位置）才需要人给一点信息，且可以只要 1 个无歧义的指认。

## 2. 本批的性质：**研究先行，不建产品件**

b3 的教训是「在未验证的前提上建了整整一批」。故本批**只回答一个问题**：

> 在我们真实的空房照片上，消失点法能否稳定地把相机 R 与 f 估到可用精度？

**本批不写任何产品代码**（不改 `apps/api` / `apps/web` 的产品路径），全部在 `scripts/spike/` 与 `scripts/calib_truth/` 下。产出是**带数字的 go/no-go**。判 GO 才立下一批做产品化。

## 3. 关键设计决策

### D1 真值夹具是前置硬条件（F001）
**无真值不开工。** b3 五轮不收敛的根因就是没有真值 —— 所有验证都在合成数据或「门没报警」上做。

真值不必完美，但必须**可量化且可证伪**。做法：在一张角点无歧义的照片上人工建立 **≥10 个**对应，用留一法（hold-out）交叉验证求参考相机 —— 留出点的重投影误差即该真值的不确定度。若不确定度本身就大，说明这张照片不适合当真值，换一张。

**PIPL**：真值夹具只存**数值**（相机参数 + 对应坐标 + 照片 sha256），照片像素绝不入 git。

### D2 依赖决策：cv2 vs 纯 numpy（F002 开工前调查）
**已查明**：本机 `cv2` 5.0.0 可用，但**不在 `apps/api/requirements.txt` 里** → 生产容器没有。

选项与代价：
- **加 `opencv-python-headless`**：线段提取（LSD）成熟稳健，但镜像增重约 90MB，且 api 容器有 **1g 内存红线**（`--workers 1` 的 OOM 护栏，见 CLAUDE.md）。
- **纯 numpy 实现**：无新依赖，但需自写 Hough / 梯度聚线，鲁棒性与调参成本高。

**本批不做决策，只做实验**：spike 阶段允许用 cv2（研究码不进生产），但报告必须回答「若要产品化，纯 numpy 能否达到同等精度」，把依赖决策的依据交给下一批。

### D3 Manhattan 假设的诚实边界
该假设在以下情形会破：弧形墙、斜墙、开放式多轴向空间、镜面/玻璃造成的虚像线段。**报告必须给出假设失效时的表现**（是拒绝出解，还是给出错解）—— 后者才是危险的，b3 的教训就是「错得看不出来」比「明确失败」更伤。

### D4 位置（剩余 3 DoF）本批不解
有了 R 和 f 之后求位置是另一个问题（需要至少 1 个已知点或已知尺寸）。**本批只验证 R 与 f**，位置留给下一批。范围收窄是刻意的。

### D5 与旧路线的关系
现有特征点/专家模式**保持原样不动**（b3 的修复全部保留、继续在线）。A 路线判 GO 后再谈如何接管或并存。**本批零产品代码改动。**

## 4. Features

| ID | 一句话 | executor |
|---|---|---|
| F001 | 真值夹具（≥10 对应 + 留一法交叉验证，产出参考相机与不确定度） | generator |
| F002 | 线段提取 + 三组正交消失点聚类（RANSAC），产出 R 与 f | generator |
| F003 | 在真值夹具上量化误差 + Manhattan 假设失效表现 → go/no-go 报告 | evaluator |

## 5. 验收总则（F003）

1. **误差量化**：在每张真值照片上给出 R 误差（角度）、f 误差（%）、以及**成功率**（多少张照片能出解）。
2. **失效表现**：Manhattan 假设不成立时，方法是**明确拒解**还是**给出错解**？给数字。这条比精度更重要 —— b3 死在「错得看不出来」。
3. **纯 numpy 可行性**：若产品化不引 cv2，精度损失多少？
4. **诚实边界**：样本量、哪些照片不适用、真值本身的不确定度。
5. **go/no-go**：明确结论 + 下一批建议（含位置求解的 3 DoF 怎么办）。
6. 红线：不改 `apps/api` / `apps/web` 产品路径；不写 `data/projects/`；PIPL 照片与像素不入 git。

## 6. 开工前调查清单（Generator 逐条实物核查后再 lock）

- F001：哪张照片的角点最无歧义（候选 `r_live` / `r_master`）；需用户配合建立对应；真值不确定度目标定多少。
- F002：cv2 LSD 在我们照片上的实际表现；纯 numpy Hough 的差距；VP 聚类的 RANSAC 参数。
- F003：R 误差多大才算「可用」—— 需反推：出图引导对相机误差的容忍度是多少（F005 spike 的引导图可复用作判据）。

## 7. 本批不做什么（防止再次范围蔓延）

- 不做位置求解（D4）
- 不做产品 UI
- 不改现有标定路径
- 不做 VLM 辅助对应（路线 B，另议）
- 不做「不依赖标定的实拍能力」（路线 C，另议，可与本批并行由用户决定）
