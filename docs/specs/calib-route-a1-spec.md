# calib-route-a1 — 从图像本身取几何（消失点 / Manhattan 约束）

> 状态：**已 lock（用户 2026-07-21 批准，§6 调查完成后）**。分支 `feat/calib-route-a1`（off b3 HEAD）→ PR → squash；绝不单推 `main`。
> 前置：`calib-cure-b3` 已收口（`docs/test-reports/calib-cure-b3-signoff-2026-07-21.md`）。
> 用户裁决（2026-07-21）：路线调整为 A。

## 1. 为什么换路线

b3 收口签收 §3 的三个结构性病根，一句话概括：

> **我们把 7 个自由度里最难的一步（像素 ↔ 抽象平面角的对应）交给了人，而人做错时系统测不出来。**

决定性数据：生产 13 份存档标定中，走新特征点模式的是 **0**；用户 4 轮真实浏览器 L2 **4 轮全败**；隔离验收实测正对单墙位形中 **26.3%** 解出「在门内但错」的相机而 reproj 接近 0。

**A 路线的核心主张：旋转与焦距应当从图像本身取，而不是从人的对应取。**

室内照片天然满足 Manhattan 假设（墙、地、天花三组互相正交）。三组正交消失点可直接给出旋转 `R` 与焦距 `f` —— 那是 7 个自由度里的 4 个，且**不依赖任何人工对应**。剩下 3 个（位置）才需要人给一点信息，且可以只要 1 个无歧义的指认。

## 2. 本批的性质：**研究先行，不建产品件**

b3 的教训是「在未验证的前提上建了整整一批」。故本批**只回答一个问题**：

> 在我们真实的空房照片上，消失点法能否稳定地把相机 R 与 f 估到可用精度？

**本批不写任何产品代码**（不改 `apps/api` / `apps/web` 的产品路径），全部在 `scripts/spike/` 与 `scripts/calib_truth/` 下。产出是**带数字的 go/no-go**。判 GO 才立下一批做产品化。

## 3. 关键设计决策

### D1 真值夹具是前置硬条件（F001）
**无真值不开工。** b3 五轮不收敛的根因就是没有真值 —— 所有验证都在合成数据或「门没报警」上做。

真值不必完美，但必须**可量化且可证伪**。用留一法（hold-out）交叉验证求参考相机 —— 留出点的重投影误差即该真值的不确定度。若不确定度本身就大，说明这张照片不适合当真值，换一张。

**PIPL**：真值夹具只存**数值**（相机参数 + 对应坐标 + 照片 sha256），照片像素绝不入 git。

#### D1.1 订正：改用「仅地面平面」（2026-07-21，用户裁决）

本节原定「在一张照片上人工建立 **≥10 个**对应」。实际执行时该方案卡死，根因是一条更根本的事实：

> **平面图只可靠地知道水平尺寸，所有竖直坐标都是假设。**
> 天花 2700 = `_REAL_CEILING_MM` 硬编码（`decor-envelope-b1-spec` §198 自承：该值来自「地面 v=1161 → 天花 v=571」的方向验证，**不是量出来的数值**，且当时明确否决了「从标定反推」）；门头 2050 = 常量；窗台高度 = geometry 里根本没有字段。
> **拿假设当世界坐标建出来的不是真值。**

实证：用户在 r_master / r_live 各标 6 点（3 组竖直配对），拟合残差 **60px**（6 点 11 自由度、超定 1 个，本该近零）；把层高放开当自由参数，DLT 反解出 3175mm、地平线比值法反解出 ~4000mm、产品硬编码 2700mm —— **三个方法三个答案**，数据不足以支撑任何一个。且 r_master 全部候选仅 12 个（8 个 z 是假设值），门在相机背后 → 上限 6~8，**结构性达不到 10**。

**订正后的做法：只用地面点（墙脚线转角 + 门洞下沿），单平面单应分解出 f/R/t。**

| | 原方案 | 订正后 |
|---|---|---|
| 世界坐标 | 混合高度，多数是假设值 | 仅 z=0；x/y 来自平面图，**零假设** |
| 标注量 | ≥10 | **≥5**（单应需 4，留一自检需 5） |
| 额外假设 | 层高 / 门头 / 窗台 | 仅「主点在图像中心」 |
| f 精度（合成实测） | — | **+0.2%** @1px 噪声，+1.0% @6px |

**自检门（三层）**：单应给出两路独立的 f 估计（正交约束、等模长约束）；每路用**留一法的变异系数**判稳定性（CV<10%），至少一路稳定，两路都稳定时须互相印证（≤15%）。

> 为什么用留一 CV 而不是条件数：实测条件数随画幅系统性漂移（4032×3024 与 1024×768 差一个量级），绝对阈值跨画幅误判 4/35；留一 CV 无量纲，实测在 4 焦距 × 3 画幅下几乎不变（正对位形恒为 42%/56%）。跨 32 组**错误采信坏解 0 次**。

**关键安全性质**：平面正对相机时**不是解不出，而是解出荒谬 f 却不报错**（实测 +3754%）。安全性不来自「算不出会报错」，而来自自检门。这正是 b3「错得看不出来」的同型病，必须由门拦。

**顺带的既有归因订正（未用真实数据验证，按用户裁决暂不据此改产品）**：calib-cure-b2 把失败归因于「特征共面 → 几何退化」。共面**不是**原因（本方案正是靠共面点工作的），**平面正对相机**才是。

### D2 依赖决策：cv2 vs 纯 numpy（F002 开工前调查）
**已查明**：本机 `cv2` 5.0.0 可用，但**不在 `apps/api/requirements.txt` 里** → 生产容器没有。

选项与代价：
- **加 `opencv-python-headless`**：线段提取（LSD）成熟稳健，但镜像增重约 90MB，且 api 容器有 **1g 内存红线**（`--workers 1` 的 OOM 护栏，见 CLAUDE.md）。
- **纯 numpy 实现**：无新依赖，但需自写 Hough / 梯度聚线，鲁棒性与调参成本高。

**本批不做决策，只做实验**：spike 阶段允许用 cv2（研究码不进生产），但报告必须回答「若要产品化，纯 numpy 能否达到同等精度」，把依赖决策的依据交给下一批。

### D3 Manhattan 假设的诚实边界
该假设在以下情形会破：弧形墙、斜墙、开放式多轴向空间，以及**正对单一平面的特写**（衣柜门、木格栅墙——线段数量可以很大，但方向只有一组）。**报告必须给出假设失效时的表现**（是拒绝出解，还是给出错解）—— 后者才是危险的，b3 的教训就是「错得看不出来」比「明确失败」更伤。

> **订正（§6 调查实测，2026-07-21）：** 本节原将「镜面/玻璃造成的虚像线段」列为破坏因素，**实测相反**——`r_live/4ae9362c` 有大面积镜面地板，仍判自洽且 f 误差 −1.1%。原因：竖直线在**水平**地面的镜像仍然竖直，反而强化垂直消失点。故该项从失效清单移除，换为实测确认的「正对单一平面的特写」（两张此类照片的 f 误差达 +347% 与 +1748%）。竖直镜面（穿衣镜、玻璃隔断）未取样，仍属未知，不得据本条推断。

### D4 位置（剩余 3 DoF）本批不解
有了 R 和 f 之后求位置是另一个问题（需要至少 1 个已知点或已知尺寸）。**本批只验证 R 与 f**，位置留给下一批。范围收窄是刻意的。

### D5 与旧路线的关系
现有特征点/专家模式**保持原样不动**（b3 的修复全部保留、继续在线）。A 路线判 GO 后再谈如何接管或并存。**本批零产品代码改动。**

## 4. Features

| ID | 一句话 | executor |
|---|---|---|
| F001 | 真值夹具（≥10 对应 + 留一法交叉验证，产出参考相机与不确定度） | generator |
| F002 | 线段提取 + 三组正交消失点聚类（RANSAC），产出 R 与 f | generator |
| F003 | 在真值夹具上量化误差 + Manhattan 假设失效表现 → go/no-go 报告 | evaluator |

## 5. 验收总则（F003）

1. **误差量化**：在每张真值照片上给出 R 误差（角度）、f 误差（%）、以及**成功率**（多少张照片能出解）。
2. **失效表现**：Manhattan 假设不成立时，方法是**明确拒解**还是**给出错解**？给数字。这条比精度更重要 —— b3 死在「错得看不出来」。
3. **自检判据必须在其未见过的照片上验证（硬性）**：§6 调查得出的自检门（三对正交 VP 均解出 f 且三值 spread < 15%）是**事后从同一批数据总结**的，属拟合而非泛化。F003 报告中，凡用于支撑成功率/GO 的数字，必须来自**判据成型后才纳入的照片**；在判据来源照片上复算出的成绩须单列并标注「拟合精度，不计入成功率」。**违反此条即等于换姿势重犯 b3 的错**（判据与被判对象同源）。
4. **R 与 f 分开判**：f 有 EXIF 可作外部真值，R 没有——而 R 占 4 个自由度里的 3 个。**GO 的门槛不得只看 f**；R 误差未量化即不得判 GO。
5. **纯 numpy 可行性**：若产品化不引 cv2，精度损失多少？
6. **诚实边界**：样本量、哪些照片不适用、真值本身的不确定度。
7. **go/no-go**：明确结论 + 下一批建议（含位置求解的 3 DoF 怎么办）。
8. 红线：不改 `apps/api` / `apps/web` 产品路径；不写 `data/projects/`；PIPL 照片与像素不入 git。

- F001：哪张照片的角点最无歧义（候选 `r_live` / `r_master`）；需用户配合建立对应；真值不确定度目标定多少。
- F002：cv2 LSD 在我们照片上的实际表现；纯 numpy Hough 的差距；VP 聚类的 RANSAC 参数。
- F003：R 误差多大才算「可用」—— 需反推：出图引导对相机误差的容忍度是多少（F005 spike 的引导图可复用作判据）。

### 6.1 调查结果（2026-07-21 完成，lock 前）

研究码在会话 scratchpad，未入 git；lock 后由 F002 正式落 `scripts/spike/`。样本 = 生产 baseline v7 的 12 张照片（其中 10 张有 `room_id`，属合法房间照）。

**外部真值的意外来源：EXIF。** 4 张 4032×3024 原图保留 EXIF（iPhone 14 Pro Max），其余 10 张 2048×1536 的 EXIF 被上传归一化剥除。据此查明两件事：

1. **f 在照片集内不是常数** —— 同一部手机混用了超广角（等效 14mm，水平 fov 102.1°）与主摄（等效 24mm，71.6°）。任何「假设 f 固定」的做法都会错。
2. **上传管线剥 EXIF，等于扔掉了一个免费的 f 真值。** 产品化时值得考虑保留（属下一批的输入，本批不做）。

**cv2 LSD 可用**（5.0.0；该算法曾在 4.1–4.5 因专利移除，5.0 已恢复）。

**裸方法从不拒解：12/12 全部"出解"**，且错解自带高 inlier 率 —— b3「错得看不出来」的病原样重现。**加自检门后可分离**：

> 自检门 = 三对正交 VP 都解出有效 f，且三值 spread < 15%

| 指标 | 数值 |
|---|---|
| 自检门接受 | 5/10 合法房间照 |
| 接受组 f 误差 | EXIF 直接真值 **−1.1%**；镜头 fov 佐证 +0.3% / +0.4% / +1.2% / −1.1% |
| 接受组种子稳定性 | 8 个随机种子 8/8 自洽，变异系数 **CV = 0.0%** |
| 拒绝组 | 8 个种子 0/8 自洽（一致拒绝），含 b2 已实证退化的 `r_guest2`、EXIF 真值错 +347% 的非房间照 |
| 边界样本 | 仅 `r_master/c2af4c4b` 一张（3/8 自洽，CV 268%） |

**⚠ 该判据属拟合，未经泛化验证** —— 见 §5.3，F003 必须在判据未见过的照片上验证，否则不得计入成功率。

**R（旋转）精度：本次完全未测。** f 有 EXIF 兜底，R 没有外部真值 —— 这正是 F001 的存在理由，也是 §5.4 的由来。

**D2 纯 numpy：未证可行。** 120 行朴素 Sobel+Hough，在单张照片上调参即过拟合：r_master 与 cv2 一致（−0.01%），但 r_cloak +12%、r_study **+241%**、r_garden **−84.5%**（真值对照）。更严重的是 r_study 上出现了**自检假阴性**（判"自洽"但错）—— 该失效模式在 cv2 前端的 10 张里一次都没出现。结论：**不是免费替换，需正经移植 LSD**；另 numpy 路径峰值内存 501MB @1024px，对 api 的 1g 红线需正视。

**F001 选片结论（用户 2026-07-21 裁决）：做两张，覆盖两枚镜头。**
- `r_master / 15877ff7…`（超广角）—— 角落机位，两墙+地+天花齐备、角点非共面，窗框/地板缝/天花凹槽给足三向线段；自检 spread 1.3% 全场最低。
- `r_live / 4ae9362c…`（主摄）—— 兼作 §D3 订正的实证样本（镜面地板）。
