## 二、逐层证据

### 1. 家具数据层 —— 无问题（关键排除项）
方案 `scheme_ai_20260714_130354_01_baec/furniture.json`：
- 书柜 `{t:bookshelf, dx:266, dy:3, w:31, h:326, z:2000}` → 世界 x 17810–18120（贴东墙 x=18150），y 2530–5790（**纵贯 3.3m 东墙**），高 2m —— 与评论"书柜铺满整个东侧墙面"**完全一致**。
- 酒柜 `{dx:-60, dy:85, w:58, h:280, z:1400}` → 贴 r_live 西墙、长 2.8m；沙发 L 形两段在客厅中南部，朝东墙电视柜（`media dx:676` 贴东墙 x≈12150）。
用户的意见与存量数据一致 ⇒ **上游数据正确，下游投影错误**。

### 2. 标定输入层 —— 根因起点
UI（`PerspectiveCalibrator.tsx`）要求：2 条"东墙(电视墙)水平边" + 2 条"南墙/落地窗水平边" + ≥2 个地面墙角（点一下 + 下拉选 NW/NE/SE/SW）。

- **798 书房**：两条 "x_lines" 实际画成**画面最右缘两段近乎竖直的短线**（(2039,1214)-(2042,1484)，(2032,82)-(2020,9)，后者仅 73px）→ 消失点完全退化；SW 锚点点在 **(6,1533) = 图像左下角像素**（真实墙角在画外，只能猜）。
- **f4d 客餐厅**：照片绑定在 `r_foyer`（1.8×3.3m 小门厅，与 r_live 同属 merge 组 m_living；**两个房间的中文标签都叫"客厅"**，极易绑错/无从分辨）。标定锚点被迫使用**门厅矩形的虚拟对角**——门厅与客厅之间是开放边界，这两个"墙角"在照片里没有任何实体特征，用户只能凭感觉点 → 解算出 z=0.4m、朝向偏差数十度的相机。
- UI 文案把"东墙=电视墙、南墙=落地窗"写死（只对客厅成立），对书房等房间是系统性误导；罗盘方位（NW/NE/SE/SW）到透视照片的映射全靠用户心算，无任何反馈。

### 3. 标定质量门 —— 全链路缺失（根因主体）
- 保存端点 `set_photo_calibration_ep`：`_reprojection_error_px` 算出误差**只存盘，不设阈值、不拒绝**（2353px 静默入库）。
- **UI 从不显示**该误差（全前端 grep 无 `reprojection`），提交成功即 toast"生成将走几何锁定精准落位"。
- 渲染端点只查 binding 指纹 stale（房间/几何变更），**不查质量**。
- 测试 `test_render_real_geometry.py:197` 断言好标定 `< 5.0px` —— **质量基准存在于测试里，从未产品化**。
- 已立项 `BL-calib-min-3-anchors`(high)：2 锚点欠约束正是本次两案的公共前提。本次事故即其生产实证。

### 4. 引导图 sanity 层 —— 覆盖面太窄
`guide_sanity_issues` 只拦"单盒覆盖 >90% 画面"一种退化。本次两案的形态——**多数盒出画**（f4d：餐桌/电视柜 in_frame=0.000，bbox 远至 x=-14998）、**盒浮空/歪斜**（798）、**相机高度 0.4m / 相机在墙里**——全部不在检查范围，顺利放行。

### 5. prompt 降级层 —— 反向放大
`box_usability` 对出画盒只做话术降级。f4d 的餐桌 `in_frame=0.000` 且 `near=True`，prompt 生成 **"sits in the near foreground… render it at full foreground scale where its box is"** —— 引导图里根本没有这个盒子，等于书面授权模型在前景自由发挥一张大餐桌。"partly outside the frame: render only its visible part" 对 0% 可见的沙发/电视柜同理（模型无从"只画可见部分"）。

### 6. 验收层 —— 对"落位错"结构性失明
- `acceptance.py`（启发式）只判 furnished/residue/reframe/structure，即"画没画/彩盒残留/背景动没动"——**设计如此，文档如实记录盲区**。f4d 背景保真 0.967 满分通过。
- `semantic_accept.py`（VLM 语义验收）**正是为此病灶建的**——其 docstring 引用的动机案例就是"生产实证: 酒柜从红盒移到左墙变嵌入式"，与本次 f4d 评论一字不差。但 `GEOMETRY_ACCEPT_VLM` **默认关，生产 .env 未开** ⇒ 同一失败模式二次发生时，为它建的防线是关着的。
- auto_check FAIL 是软门：798 两次尝试均 FAIL 仍交付最佳图。（透出徽标已有，但"精准落位"品牌下交付一张已知重绘 81% 的图，用户预期落差大。）

### 7. 其他观察（非根因）
- `requested 1536x1024 (3:2)` vs `actual 1448x1086 (4:3)`：relay 输出跟随输入 4:3 纵横比，对齐无害；`pick_edit_size` 对 4:3 照片选 3:2 档略欠优，仅记录。
- `direction`(v0..v3，拍摄视角→轴测旋转) 是**既有但未被利用**的相机朝向交叉校验信号。
- 渲染 engine_version=d6d6506（decor-envelope-b1），与当前生产 8e645e4 间无本区域行为差异。

## 三、根因链（一句话版）

**标定 UX 容易产出错误输入（虚拟角点/方位映射/写死文案）→ 坏位姿带着 2353px/112px 的自报误差畅通入库（无门禁、不透出）→ 盒子投歪/投丢，窄 sanity 拦不住 → 出画盒被降级话术反向授权模型自由发挥 → 背景保真的验收对落位错失明，为此建的 VLM 层生产未开 → FAIL 软门照常交付 → 用户看到"精准落位"出的错位图。**

## 四、修复建议（分层，待立项）

**P0 止损（便宜、确定性、不调 AI）**
1. **标定质量硬门**：保存时 reprojection_error 超阈值（建议 ≤ 图幅对角线 1%，即 2048×1536 下 ~25px；好标定实测 <5px）→ 400 拒绝并提示重标；渲染入口对存量标定同样检查（拦 photos.json 里的坏存量）。
2. **相机物理 sanity**：解算相机高度 z ∉ [800, 2200]mm，或相机位置不在绑定房间（merge 组并集外扩 ~1m）内 → 拒绝。798（z=1370 但在墙外）与 f4d（z=399）都会被拦。
3. **guide 健全性扩展**：locked 家具中 `in_frame_frac < 0.15` 的比例超 1/3 → `DEGENERATE_GUIDE` 阻断（复用现有 409 门禁码，零成本）。
4. **矛盾话术禁止**：`near=True` 且 `in_frame≈0` 时不得生成"前景全尺寸"指令。
5. **UI 透出**：标定提交后显示误差与质量评级（好/可用/不可用），把测试里的 <5px 经验值产品化。

**P1 治本（标定 UX）**
6. **锚点 ≥3 + 实时反投预览**：点完锚点立即把房间线框/地脚线用解算相机反投叠加到照片上——用户一眼看出歪斜，等于零成本人工验收环（最高杠杆单项）。`BL-calib-min-3-anchors` 升级为本批执行。
7. **merge 组房间的锚点源**：改用 merge 组外包络角点或任选可见实体墙角（下拉列出组内全部角点），禁止开放边界虚拟角；照片绑定 UI 对同名房间（两个"客厅"）显示区分徽标。
8. **direction 交叉校验**：解算相机 yaw 与用户标注的 v0..v3 象限粗比对，反向即拒。

**P2 兜底（验收）**
9. 生产开启 `GEOMETRY_ACCEPT_VLM`（评估每图 +2 次 chat 成本）；它检查的就是本次的失败模式。
10. auto_check FAIL 的交付语义复议：至少在 UI 大幅降权/默认折叠，或要求用户显式确认后才计入可用产出。
