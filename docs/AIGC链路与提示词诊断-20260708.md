# AIGC 链路与提示词诊断报告

- 日期: 2026-07-08
- 范围: 本仓库 AIGC 功能链路、提示词质量、image2 真实生图质量与后续优化建议
- 约束: 本次只做诊断、测试与报告落盘, 不修改业务代码
- 凭据处理: 真实测试使用用户临时授权的 OpenAI-compatible relay, API Key 未写入仓库、报告或 `.env`

## 一句话结论

当前 AIGC 链路的工程骨架是健康的: AI 方案生成、轴测底图、image2 单图编辑、空房图上传、多图实拍编辑、预算、异步任务、产物归档和提示词溯源都能跑通。

真实 provider 测试进一步确认: D 户型在 `gpt-image-2` 上可以产出可用级效果图, 轴测图视觉完成度高, 实拍图对房间透视、窗墙结构和家具落位保持较好。但限制 image2 质量上限的核心问题仍在提示词与输入图组织: 第 7 步实拍 prompt 没有贯通方案 `style_prompt`, 照片用途没有在实拍页/后端过滤, 软装小件仍缺少结构化描述, 且 provider 返回图尺寸可能与请求尺寸不一致但当前记录未校验。

## AIGC 链路梳理

### 第 2 步: AI 软装方案

入口: `POST /api/projects/{house}/furnish`

关键实现:

- `apps/api/main.py:1293` 进入 `/furnish`。
- `apps/api/main.py:1303` 强制要求非空 `style_prompt`。
- `apps/api/main.py:1328` 明确 AI 不重新落位, 只基于锁定布局做风格候选。
- `apps/api/main.py:1333` 调 `furnish_service.generate_candidates(...)`。
- `apps/api/main.py:1352` 将候选 `style_prompt` 写入方案 meta。

提示词定位是合理的: 当前 AI 不负责几何摆放, 只输出方案名、丰富风格描述和可选同组换件, 能降低家具漂移和非法坐标风险。

主要不足:

- `style_prompt` 仍是自由文本, 没有结构化字段, 后续难以稳定控制硬装、软装、灯光、材质、禁忌和房间优先级。
- 方案生成后 `style_prompt` 不易编辑迭代, 用户发现出图偏差后无法在现有 UI 中细调方案风格。

### 第 5 步: 轴测 image2 出图

入口: `POST /api/projects/{house}/schemes/{scheme_id}/render-ai`

关键实现:

- `apps/api/main.py:1436` 进入 `_render_ai_response`。
- `apps/api/main.py:1467` 渲染 photo 模式轴测 SVG。
- `apps/api/main.py:1470` 根据 SVG 比例选 image2 输出档位。
- `apps/api/main.py:1471` 栅格为 PNG, 作为单图 edits 输入。
- `apps/api/main.py:1474` 读取方案 `style_prompt`。
- `apps/api/main.py:1476` 调 `prompt_gen.generate(..., style=style)`。
- `apps/api/main.py:1497` 调 provider `edit(prompt, [base_png], size=...)`。
- `apps/api/main.py:1550` 记录请求尺寸。
- `apps/api/main.py:1554` 归档 prompt 原文。

提示词质量整体较好:

- 明确锁定相机、几何、墙、窗、家具位置。
- 按房间输出家具与材质。
- 方案 `style_prompt` 已贯通到 prompt 头部。
- with_positions 会给家具补充房内方位, 有利于减少漂移。

主要不足:

- `packages/floorplan_core/floorplan_core/prompt_gen.py:119` 仍跳过 `partition` 和 `rug`。地毯是影响室内效果图质感的重要软装, 不应被布局级提示词静默漏掉。本次真实测试中轴测 prompt 出现“地毯”, 是因为 AI 风格文本里写了地毯材质; per-room 家具清单仍没有描述 `rug`。
- 目前 prompt 是一整段长文本, 缺少稳定的分层结构, 对模型来说“硬约束、软风格、每房清单、负面约束”的权重不够清晰。

### 第 7 步: 空房照 + 轴测参考实拍渲染

- `apps/api/main.py:1901` 多图 edits 输入顺序为: 空房图、轴测参考图、墙面参考图。

这条链路的架构方向是正确的: 第一张图锚定真实结构, 第二张图提供家具方案, prompt 强调保持相机、结构、门窗、自然光和大件靠墙。

主要不足:

- 第 7 步 prompt 没有接收方案 `style_prompt`; 风格主要依靠第二张轴测参考图隐式传递。这在真实测试中能产出不错结果, 但不同风格方案的可控性不稳定。

## 新增问题清单

### P0: 实拍照片用途未过滤

现象:

- `PHOTO_PURPOSES` 已区分 `empty`、`wall_material`、`underlay`, 见 `apps/api/baselines.py:613`。
- 墙面材质选择器会过滤 `purpose === "wall_material"`。
- 但实拍效果图页面加载并展示全部 baseline photos, 后端 `render-real` 也只按 `photo_id` 查找, 不校验 `purpose == "empty"`。

影响:

- 墙面材质图、底图描摹图或其他非空房照可能被误选为实拍底图。
- 一旦误选, image2 会把错误图片作为第一结构锚点, 高概率产出不可用结果并消耗额度。

建议:

- 前端 `real-render` 页面只展示 `purpose === "empty" || purpose == null` 的照片。
- 后端 `render-real` 对 `photo.purpose` 做硬校验, 非 `empty/null` 直接 400。
- 文案和接口字段应把 `null` 历史照片视为旧版 empty, 新上传默认显式写 `empty`。

### P0: 第 7 步实拍 prompt 未贯通方案风格

现象:

- 第 5 步 `_render_ai_response` 会读取 scheme meta 的 `style_prompt` 并传给 `prompt_gen.generate`。
- 第 7 步 `_real_render_prompt(photo, furniture, G, scope=...)` 没有 `style` 参数。
- 真实测试中 `render_real` prompt 不包含“现代轻奢 / 胡桃木”等方案风格词。

影响:

- 实拍图风格主要靠第二张轴测参考图隐式传递。
- 如果轴测参考本身风格弱、参考图被 letterbox 后家具占比低, 或用户希望在同一照片上比较多种软装风格, 第 7 步可控性会下降。

建议:

- `_real_render_prompt` 增加 `style_prompt` 输入。
- prompt 中区分硬装保护和软装风格: 保持第一张照片的建筑结构、门窗、透视、自然光; 允许按方案风格替换可移动软装、家具材质、窗帘、地毯、灯具、摆件。
- 对空房图已有硬装的场景, 明确“不要改变固定硬装材质, 仅改变可移动家具和软装”; 对毛坯/半硬装图, 允许按方案补齐软装但不改墙体。

### P1: provider 返回尺寸未校验

现象:

- 第 7 步请求尺寸记录为 `1536x1024`, 真实返回图为 `1677x938`。
- record 的 `size` 字段仍保存请求尺寸, 未保存实际宽高。

影响:

- 产物元数据不准确。
- 后续做 A/B 对比、像素级评估、前端预设画布、下载规格控制时会出错。

建议:

- provider 返回后用 Pillow 读取实际宽高。
- record 增加 `requested_size`、`actual_size`。
- 如业务要求固定输出档位, 保存前统一 letterbox/crop/resize 到请求尺寸; 如允许 provider 自适应, UI 和报告应展示实际尺寸。

### P1: rug 被布局级 prompt 跳过

现象:

- `prompt_gen.py:119` 跳过 `rug`。
- 本次真实测试中 prompt 出现“地毯”只是来自 AI 风格描述, per-room 家具清单没有 rug。

影响:

- 地毯对空间尺度、色彩、层次和沙发区完整度影响很大。
- 缺少布局级地毯会降低客厅、卧室等空间的软装完成度和可控性。

建议:

- 不再跳过 `rug`; 对 rug 单独生成 “area rug under sofa/coffee table” 之类描述。
- 若担心模型把地毯当实体家具遮挡路径, 可归入 soft furnishings 层, 不与大件家具同权重。

### P1: 风格缺少可迭代入口

现象:

- AI 方案创建后 `style_prompt` 被写入 meta, 但当前编辑入口主要是方案名/家具, 缺少面向出图的风格调参入口。

影响:

- 用户发现“太豪华 / 太暗 / 木色不对 / 地毯不喜欢 / 更像样板间”后, 不能直接在现有方案上迭代 prompt。

建议:

- 方案详情增加“出图风格”编辑区。
- 将自由文本拆成结构化字段: 风格方向、主材、主色、点缀色、灯光色温、软装密度、禁忌、房间重点。
- 每次出图记录使用的 style snapshot, 保证历史可复现。

## image2 质量优化建议

### 1. 从自由文本升级为 Prompt Compiler

建议将输入拆成结构化 schema:

```json
{
  "style": "现代克制轻奢",
  "palette": ["暖白", "胡桃木", "米白织物", "低饱和焦糖", "墨绿色点缀"],
  "materials": {
    "fixed": ["暖白石材", "胡桃木饰面", "微水泥"],
    "soft": ["亚麻", "羊毛地毯", "皮革", "纱帘"]
  },
  "lighting": "2700K-3000K warm cove lighting, soft daylight",
  "must_keep": ["camera", "walls", "windows", "doors", "structural openings"],
  "negative": ["do not move walls", "no extra rooms", "no text labels", "no floating furniture"]
}
```

然后由 compiler 输出第 5 步和第 7 步不同权重的 prompt。

### 2. 第 5 步和第 7 步用不同 prompt 策略

第 5 步轴测图:

- 目标是“整套方案的高完成度表达”。
- 可以更强调材质、灯光、家具完整度、软装层次。
- 仍必须强锁几何、相机、房间位置。

第 7 步实拍图:

- 目标是“真实照片中摆入方案家具”。
- 第一优先级是保持实拍结构和相机。
- 第二优先级是按轴测参考落位。
- 第三优先级才是方案风格。
- 风格应该限定在可移动家具、窗帘、地毯、灯具、挂画、摆件, 不应随意改真实硬装。

### 3. 建立 image2 回归评测集

建议固定一组最小评测集:

- D 户型整宅轴测 2 张: 默认方案、AI 风格方案。
- 客餐厅实拍 4 张: 不同拍摄方向各一张。
- 主卧实拍 2 张。
- 墙面材质参考图 2 组。

每次提示词或链路调整后记录:

- provider 是否成功。
- 请求尺寸和实际尺寸。
- prompt 长度和 token usage。
- 家具落位评分。
- 结构保持评分。
- 风格符合度评分。
- 不可接受错误: 改墙、改窗、错房间、家具漂浮、人物/文字、水印。

### 4. 产物元数据补强

建议 record 中补充:

- `requested_size`
- `actual_size`
- `input_images`: 每张输入图的 purpose、room_id、direction、sha256、width、height
- `prompt_version`
- `style_snapshot`
- `provider_latency_s`
- `quality_review`: 人工或自动评分

这样后续才能把“哪类 prompt 真的更好”沉淀成可比较数据。

## 优先级建议

1. P0 修复实拍照片用途过滤: 前端过滤 + 后端硬校验, 避免误消耗额度。
2. P0 把 `style_prompt` 贯通到第 7 步实拍 prompt, 但要明确只影响软装与可移动家具。
3. P1 增加 provider 返回图尺寸校验和 `actual_size` 记录。
4. P1 让 `rug` 进入布局级 prompt, 并逐步补充窗帘、灯具、挂画、绿植、摆件等软装层。
5. P1 增加方案 `style_prompt` 编辑/版本化入口。
6. P2 建立固定 image2 回归评测集, 每次 prompt 调整都用同一批输入图横向比较。

## 最终判断

本仓库 AIGC 链路不是“不能用”的状态, 而是已经具备真实生产调用能力。真实测试证明 `gpt-image-2` 能生成质量可接受的 D 户型轴测图和客餐厅实拍效果图。

下一阶段提升质量的关键不是盲目换模型, 而是把用户意图、方案风格、房间软装、照片用途和输出规格组织得更清楚。尤其第 7 步需要从“空房照 + 轴测图 + 短 prompt”升级为“空房结构硬锁 + 方案风格软锁 + 家具落位说明 + 输入图用途校验 + 产物尺寸校验”的稳定链路。
