# AIGC 链路与提示词诊断报告

- 日期: 2026-07-08
- 范围: 本仓库 AIGC 功能链路、image2 图生图调用前后的数据准备、提示词合理性与优化方向
- 约束: 本次只做诊断分析与建议, 不改业务代码

## 一句话结论

当前 AIGC 链路的工程骨架是健康的: 方案家具、轴测底图、空房照片、墙面材质参考图、预算、异步任务、产物归档和提示词溯源都已经形成闭环。确定性测试全部通过。

真正限制 image2 输出质量的主要问题不在调用链是否能跑, 而在用户意图与软装细节传递不充分: 第 7 步实拍提示词没有贯通方案 `style_prompt`, 软装细节如地毯在第 5 步提示词中被跳过, 照片用途未过滤可能导致错误图片被当作空房照, 且当前风格输入仍是一次性自由文本, 缺少结构化约束和可迭代优化入口。

## 链路总览

### 1. AI 生成风格候选

入口:

- 后端: `apps/api/main.py` 的 `POST /api/projects/{house}/furnish`
- 服务: `apps/api/furnish.py`
- 前端: `apps/web/src/app/studio/projects/[id]/scheme/page.tsx`

流程:

1. 用户输入 `style_prompt` 与候选数量。
2. 后端读取 base scheme 的锁定家具布局。
3. LLM 只输出候选方案名称、富化后的 `style_prompt`、可选同组换件 `swaps`。
4. 后端校验 `swaps` 必须来自同一 `swap_group`, 再确定性套回原家具坐标。
5. 新建多个 scheme, 不覆盖原方案。

诊断:

- 功能实现合理。当前已经从早期“AI 自动落位”收敛到“人工锁定布局 + AI 风格候选/同组换件”, 这更符合产品现实。
- `build_messages()` 中明确禁止输出坐标、尺寸或解释, 这一点正确。
- 主要不足是 LLM 输出的 `style_prompt` 仍是自由文本, 后续不可结构化检查, 也难以稳定映射到 image2 的材质、灯光、单品细节。

### 2. 轴测效果图生成

入口:

- 后端: `apps/api/main.py` 的 `POST /render-ai`
- 提示词: `packages/floorplan_core/floorplan_core/prompt_gen.py`
- 渲染: `floorplan_core.axon`

流程:

1. 读取 scheme 绑定的 baseline geometry 和 furniture。
2. `axon.build_scene()` 生成 canonical scene, 校验通过后继续。
3. 渲染 photo 轴测 SVG。
4. 根据 SVG viewBox 选择 image edit 输出比例, 并 letterbox 成精确 PNG 画布。
5. 生成逐房提示词, 注入 `style_prompt`, 开启房内方位短语。
6. 调 `provider.edit(prompt, [base_png], size=..., model=image2)`。
7. 保存成品、底图、缩略图、预览图、prompt 原文、scene manifest、usage。

诊断:

- 功能实现正常, 且旧审计里的多个 P0 问题已经修复: 输入/输出比例对齐、`style_prompt` 贯通第 5 步、底图与 prompt 归档、渲染记录瘦身、缩略图/预览图、自托管产物。
- 提示词整体方向正确: 强锁相机、几何、墙、窗、家具位置、尺寸和朝向, 只做材质化和灯光化。
- 风险在于 prompt 对“软装精致度”的表达不足。它能告诉 image2 有哪些家具和在哪, 但对窗帘、地毯、床品、抱枕、灯具、饰品、材质档次的表达偏弱。

### 3. 空房实拍效果图生成

入口:

- 后端: `apps/api/main.py` 的 `POST /render-real`
- 前端: `apps/web/src/app/studio/projects/[id]/real-render/page.tsx`

流程:

1. 选择 baseline 下的照片。
2. 读取照片字节作为第一张 image edit 输入。
3. 按照片标注的 `room_id` 切片轴测图, 按 `direction=v0..v3` 旋转视角。
4. 追加墙面材质参考图, 最多满足 provider 的 `MAX_EDIT_IMAGES=4`。
5. 生成 `_real_render_prompt()`。
6. 调 `provider.edit(prompt, [empty_photo, axon_ref, wall_refs...])`。
7. 保存成品、轴测参考底图、prompt、照片指纹、方向、墙面参考图 id。

诊断:

- 工程链路比早期文档所描述的状态成熟很多: 照片会归一化为 JPEG, EXIF 方向被物化, GPS 被剥离, 输出尺寸按照片比例选择, 轴测参考按房间切片并可按视角旋转。
- 实拍提示词对“保持第一张照片的真实结构、门窗、材质、透视、自然光不变”表达明确, 这是正确的。
- 主要问题是风格和单品细节没有充分进 prompt。它目前主要依赖第二张轴测图承载家具方案, 但 image2 对具体 SKU/材质/软装氛围的保持能力有限, 纯图像参考容易漂。

### 4. Provider、预算、异步与产物

相关文件:

- `apps/api/aigc/providers.py`
- `apps/api/aigc/budget.py`
- `apps/api/aigc/jobs.py`
- `apps/api/aigc/artifacts.py`
- `apps/api/aigc/imaging.py`
- `apps/api/aigc/raster.py`

诊断:

- provider 抽象清晰, 单图使用 `image`, 多图使用 `image[]`, 并按 magic bytes 判别 PNG/JPEG/WEBP。
- 预算用预扣模型, 图像生成失败释放, chat furnish 计次不释放, 逻辑可控。
- job 是进程内异步, 适合当前单 worker MVP。未来多 worker 或重启恢复需要 Redis/RQ 等外部队列。
- 产物只服务位图扩展名, SVG 不进 artifacts, 安全边界合理。

## 提示词诊断

### furnish prompt

位置: `apps/api/furnish.py`

当前优点:

- 角色清楚: 室内软装风格助理。
- 任务边界正确: 不移动家具、不增删、不输出坐标。
- 输出结构受控: JSON object, schemes 数组, name/style_prompt/swaps。
- 后端二次校验有效: 房间、源件、同组换件都做白名单过滤。

问题:

- `style_prompt` 富化没有结构化 schema, 只能靠自然语言质量。
- 没有明确要求 `style_prompt` 分层包含材质、主色、点缀色、灯光、软装、禁忌。
- 候选差异只靠“明显不同的风格方向”一句话, 可重复性和可控性一般。
- 换件只按同组约束, 但没有风格标签/尺寸适配/房间语境约束, 风格合理性主要靠 LLM 自觉。

建议:

- 将 `style_prompt` 拆成结构化字段: `palette`, `materials`, `lighting`, `soft_decor`, `room_overrides`, `avoid`。
- LLM 输出结构化字段后, 后端再编译成 image2 prompt。
- 增加候选去重标准, 例如不同候选必须在主材、主色、灯光氛围中至少两项不同。

### axon prompt

位置: `packages/floorplan_core/floorplan_core/prompt_gen.py`

当前优点:

- 强锁 camera、geometry、walls、windows、furniture position/size/orientation。
- 每房列出家具与基础材质。
- `with_positions=True` 时会把家具方位加入 prompt, 且使用 axon 调整后的坐标, 与底图一致。
- `style_prompt` 已经注入第 5 步。
- 墙面材质标签能进 prompt; 若墙面挂了实拍参考图, 语义材质短语会让位, 避免双重信号冲突。

问题:

- `rug` 已在目录和轴测图里渲染, 但在 prompt 里被跳过。当前 D 方案有 2 张地毯, 自动 prompt 不提地毯。
- prompt 的家具描述仍偏“类型级”, 缺少“材质/颜色/款式/档次”。例如 `a sofa` 不足以还原“焦糖皮/墨绿丝绒/奶咖布艺”等设计意图。
- 默认材质词表硬编码较多, 与早期软装设计文档中的丰富描述相比退化明显。
- 负向约束不如早期手工提示词完整, 例如“不要改变地板/墙面/吊顶/门窗/窗外景观/中央空调/橱柜/硬装”的约束在实拍链路里更明确, 在轴测链路里偏概括。

建议:

- 不要跳过 `rug`, 至少输出 `a warm taupe area rug under/around ...` 这类表达。
- 对 `catalog.CATALOG[t]` 增加可选 `style_tags` 或 `render_prompt` 字段, 让同一家具类型可以在不同风格下生成不同材质描述。
- 将早期手工提示词中的高价值软装元素沉淀成风格库: 窗帘、地毯、抱枕、床品、灯具、艺术品、绿植、金属件、色温。
- 对第 5 步增加负向约束片段, 明确 no text/no labels/no warped walls/no extra furniture/no missing furniture。

### real-render prompt

位置: `apps/api/main.py` 的 `_real_render_prompt()`

当前优点:

- 第一张图是结构锚点, 第二张图是方案参考图, 输入顺序正确。
- 明确保持第一张照片结构、门窗、墙地材质、透视、自然光不变。
- 针对床、沙发、衣柜、电视柜等大件, 有“不贴玻璃/不悬空/背靠实墙”的规则。
- 若照片标注了视角, 会提示轴测参考已旋转对齐, 并补充相机相对落位。

问题:

- 没有读取 scheme 的 `style_prompt`, 所以用户风格意向不会进入实拍效果图 prompt。
- 没有按房间输出具体材质和软装描述, 只说“按照轴测参考图中的家具布局、款式与配色”。
- 对 wall material 参考图的解释不足。当前只是把墙面图作为第三/第四张输入, prompt 没有明确告诉模型“后续参考图仅用于墙面材质, 不要复制其构图/家具/视角”。
- 如果用户未标注房间, 会回退整宅轴测参考, 这是质量最差路径。前端已有提示, 但仍允许生成。

建议:

- 将 `style_prompt` 注入 `_real_render_prompt`, 但要声明“仅影响新增家具、软装、灯光氛围, 不改变原照片硬装”。
- room scope 下增加逐房家具短清单, 至少包含每件类型、中文名、材质/颜色、相机相对落位。
- 对墙面参考图增加专门说明: “第三/第四张图只作为墙面材料纹理参考, 不复制其中家具、透视或空间结构。”
- 对未标注 `room_id` 或未选 `direction` 的照片, 建议从“允许但提示”升级为“默认阻断, 用户可强制继续”。

## 功能风险清单

### P0: 照片用途未过滤

现象:

- `BaselinePhoto.purpose` 支持 `empty`, `wall_material`, `underlay`。
- 墙面材质控件会上传 `purpose=wall_material`。
- 底图描摹控件会上传 `purpose=underlay`。
- 实拍生成页 `listBaselinePhotos()` 后没有过滤, 后端 `render-real` 也只按 `photo_id` 查找, 不校验 purpose。

影响:

- 用户可能把墙面局部材质图或户型底图误选为空房照。
- image2 会把错误图片当结构锚点, 结果必然错。

建议:

- 前端实拍页只展示 `purpose == null || purpose == "empty"` 的照片。
- 后端 `render-real` 校验 `purpose` 也必须为空或 `empty`。
- 墙面材质参考收集时反向校验 `purpose == "wall_material"`。

### P0: 第 7 步 style_prompt 断链

现象:

- 第 5 步 `render-ai` 会读取 scheme meta 的 `style_prompt`。
- 第 7 步 `render-real` 不读取 scheme style。

影响:

- 同一方案的轴测效果图和实拍效果图可能风格不一致。
- 用户在方案中心输入的“现代轻奢/原木/奶油风”等需求不能稳定影响实拍图。

建议:

- `_real_render_prompt(photo, furniture, G, scope, style=None)` 支持 style。
- `_render_real_response()` 读取 `_scheme_meta2["style_prompt"]` 并传入。

### P1: `rug` 被提示词跳过

现象:

- catalog 中 `rug` 已升格入目录。
- axon 2D/3D 已能渲染 rug。
- `prompt_gen.generate()` 仍 `if it["t"] in ("partition", "rug"): continue`。

影响:

- 地毯是提升室内效果图质感的关键软装, 但 image2 没有明确文字约束。
- 仅靠底图里的薄片/线框, 模型可能忽略或误解地毯。

建议:

- 只跳过 `partition`, 不跳过 `rug`。
- 根据房间与风格编译地毯描述, 如 `a warm taupe wool area rug anchoring the sofa group`。

### P1: 风格不可编辑

现象:

- `schemes.patch_scheme()` 目前只允许修改 `name`。
- 前端展示 `style_prompt`, 但没有编辑入口。

影响:

- 用户后续想微调风格, 只能重新生成 AI 候选或手改文件。
- 已编辑过家具的方案无法低成本更新出图风格。

建议:

- 允许 draft scheme 修改 `style_prompt`。
- 增加“本次出图补充要求”, 与 scheme 长期风格分开。

## 对 image2 的优化建议

### Prompt 编译器而不是纯字符串拼接

建议建立统一的 prompt 编译层, 输入结构化数据:

```json
{
  "style": {
    "name": "现代轻奢",
    "palette": ["oatmeal", "walnut", "caramel leather", "dark green velvet", "brushed brass"],
    "lighting": "warm 2700K cove lighting, soft daylight",
    "avoid": ["cold stainless steel", "over-saturated colors"]
  },
  "room": {
    "id": "r_live",
    "name": "餐厅区",
    "hard_finishes": ["beige travertine floor", "off-white walls"],
    "furniture": [
      {"type": "dining_table", "material": "walnut and stone", "placement": "center"},
      {"type": "rug", "material": "warm taupe wool", "placement": "under seating group"}
    ]
  },
  "mode": "axon-photoreal | real-photo"
}
```

再分别编译为第 5 步和第 7 步 prompt。这样后续可以做 schema 校验、A/B、版本化和回放。

### 增加分层约束

推荐 prompt 层级:

1. 输入图片角色: 第一张/第二张/第三张各自是什么。
2. 不可变约束: 结构、硬装、透视、门窗、墙地、相机。
3. 可变范围: 只新增/材质化家具、软装、灯光氛围。
4. 房间/家具清单: 类型、数量、位置、材质、颜色。
5. 风格: palette、materials、lighting、decor。
6. 负向: no text, no labels, no people, no extra walls, no changed windows, no missing furniture。

### 建立 image2 质量评估集

建议固定一套 eval:

- 3 个典型房间: 客厅/主卧/卫生间。
- 每个房间 2 张空房照: 横拍/竖拍。
- 3 种风格: 现代轻奢/原木日式/奶油风。
- 评估维度:
  - 结构保持: 门窗、墙面、相机是否变。
  - 家具完整: 关键家具是否缺失/多出。
  - 落位正确: 大件是否靠对墙、是否挡门、是否贴窗。
  - 风格一致: 主色、材质、灯光是否符合 style。
  - 视觉质量: 写实度、比例、阴影、材质可信度。

每次改 prompt 后用同一输入跑小批量, 以人工评分或多模态评审记录版本差异。

## 优先级建议

短期优先:

1. 过滤照片用途, 防止错误图片进入第 7 步。
2. 第 7 步 prompt 注入 scheme `style_prompt`。
3. 第 5 步 prompt 恢复 rug。
4. 增加 wall material 参考图的 prompt 说明。
5. 提供单次出图补充 prompt/negative prompt 入口。

中期:

1. 将 `style_prompt` 结构化。
2. 建立风格库与房间级材质库。
3. 扩展 catalog 的 image prompt 字段。
4. 建立 image2 eval 集。

长期:

1. 多 worker 持久任务队列。
2. 更强的多图参考策略, 例如单品参考图、材质参考图、房间参考图分槽说明。
3. 对重要房间支持分阶段生成: 先大件, 再软装细节, 最后灯光和饰品。

## 最终判断

当前链路已经具备产品化基础, 且关键工程防线较完整。下一阶段不建议继续大改底层调用链, 应把重点放在“提示词编译”和“用户需求结构化”上。

对 image2 来说, 最有价值的改进不是再加一句泛泛的“更高质量”, 而是把现有设计资料里的可执行信息稳定传进去: 每个房间的硬装不可变约束、软装材质、主色与点缀色、灯光色温、地毯窗帘床品饰品、以及哪些东西绝对不能改。这样才能让模型既不乱改结构, 又能产出更像真实室内设计交付图的结果。
